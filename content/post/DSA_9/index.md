# 数据结构与算法笔记（九）：词典

> 这一章看似内容很多，实际可以用一句话串起来：  
> **当关键码不再适合按大小顺序组织时，我们就转向“根据对象本身直接访问”；而当仍想保留有序查找能力、又希望实现更简单时，则可以借助概率平衡的跳转表。**

## 1. 从“搜索树”走向“词典”

### 1.1 前面几章解决了什么

在前面的 BST、AVL、红黑树等结构里，我们做的事情几乎都是：

- 元素之间有**大小顺序**
- 可以比较 `< / > / =`
- 因而可以通过“左小右大”的方式组织数据
- 进而支持 `search / insert / remove`

但这类结构有一个隐含前提：

> **关键码必须适合比较大小。**

这在很多实际应用中并不自然。  
例如：

- 名字 `"关羽"`、`"张飞"`、`"赵云"`  
- 电话号码对应联系人  
- 字符串到对象的映射  
- URL、用户名、词条名

这些数据更自然的访问方式不是：

- “第几个”
- “大小次序”

而是：

- “就按这个对象本身去访问”

---

### 1.2 联合数组：按对象直接访问

课件第 2 页用“联合数组（Associative Array）”引入这一点：fileciteturn11file0

- 普通数组：下标是整数
- 联合数组：下标可以直接是对象值本身，例如字符串

例如：

```python
style["关羽"] = "云长"
style["张飞"] = "翼德"
style["赵云"] = "子龙"
style["马超"] = "孟起"
```

这里下标已经不再是：

- `0, 1, 2, 3`

而是：

- `"关羽"`, `"张飞"` 等对象值本身

于是访问方式就从：

> **循秩访问（call-by-rank） / 循关键码比较访问（call-by-key）**

进一步走向了：

> **循对象访问（call-by-object）**

这正是词典结构的核心动机。

## 2. 词典（Dictionary / Map）的抽象

### 2.1 entry = (key, value)

课件第 3 页先给出词典最基本的组成单位：fileciteturn11file0

\[
entry = (key, value)
\]

也就是说，词典中的每个元素都不再只是一个孤立值，而是一个**键值对**：

- `key`：用来查找
- `value`：附着在 key 上的实际信息

例如：

- `"翼德" -> "张飞"`
- `"中岳" -> "嵩山"`
- `"P" -> "parsley"`

---

### 2.2 Map / Dictionary 的接口

课件第 3～4 页把词典定义为“词条的集合”，并给出接口：fileciteturn11file0

- `size()`
- `put(key, value)`
- `get(key)`
- `remove(key)`

伪代码接口大致为：

```cpp
template <typename K, typename V>
struct Dictionary {
    virtual int size() = 0;
    virtual bool put(K, V) = 0;
    virtual V* get(K) = 0;
    virtual bool remove(K) = 0;
};
```

这是一个非常标准的 ADT 设计。

---

### 2.3 与 BST / PQ 的区别

课件第 3 页特别强调了两点：fileciteturn11file0

1. 关键码**不必可比较大小**
2. 因而词典中可容纳的数据类型比 BST 更广

同时，它和优先级队列（PQ）也不同：

- PQ 只关心最大 / 最小
- Dictionary 关心任意指定 key 的取值

所以从功能上看：

> **Dictionary 是比 BST / PQ 更通用的映射抽象。**

## 4. 从电话簿问题理解散列表

## 4.1 电话簿的朴素方案

课件第 10～11 页用电话簿举例：fileciteturn11file0

需求是：

> 根据电话号码，快速找到“教员 / 学生 / 员工 / 办公室”等记录。

最暴力的办法是：

- 用号码本身直接作为数组下标

查找当然是 `O(1)`，但会有巨大浪费。

以清华 2003 年为例：

- 可能的电话号码数量：

\[
R = 10^8 = 100M
\]

- 实际存在的号码数量：

\[
N = 25,000 = 25K
\]

如果真的开一个长度为 `10^8` 的数组，那么：

- 时间很快
- 但空间利用率只有：

\[
25K / 100M = 0.025\%
\]

课件第 11 页用这组数字非常形象地说明了：

> **“直接编址”虽然快，但可能极度浪费空间。**

---

## 4.2 散列表的基本思想

课件第 12 页给出散列表的核心概念：fileciteturn11file0

- 维护一个长度为 `M` 的桶数组（bucket array）
- 其中每个桶直接存放或间接指向一个词条
- 通过散列函数 `hash(key)` 把 key 映射到桶地址

要求满足：

\[
N < M \ll R
\]

也就是说：

- 散列表容量要明显大于现有元素数 `N`
- 但远小于原始关键码空间 `R`

于是空间从“接近 `R`”降到“接近 `N`”，  
而查找仍希望保持接近常数时间。

---

### 4.3 什么叫“直接”

课件第 12 页还特别写了一句很容易被忽略的话：fileciteturn11file0

> “直接”：expected-`O(1)` ≠ `O(1)`

这句话非常重要。

因为散列访问虽然常说“常数时间”，但更准确地说是：

- **期望常数时间**
- 或**分摊常数时间**

不是绝对、最坏情况意义下的严格 `O(1)`。

其能否高效，关键取决于：

1. 散列函数是否好
2. 装填因子是否合理
3. 冲突排解策略是否合适

---

## 5. 冲突：为什么无法彻底避免

## 5.1 Synonym / Collision

课件第 15 页定义了冲突：fileciteturn11file0

若：

\[
key_1 
e key_2
\quad 	ext{但} \quad
hash(key_1) = hash(key_2)
\]

则称这两个 key 是同义词（synonym），发生了 collision。

这几乎是散列表不可避免的命运。  
因为只要：

- 关键码空间很大
- 桶数 `M` 有限

根据抽屉原理，不同 key 映射到同一桶是迟早的事。

---

## 5.2 装填因子（load factor）

课件第 16 页定义：

\[
\lambda = N / M
\]

其中：

- `N`：当前词条数
- `M`：桶数

装填因子越大：

- 空间利用率越高
- 但冲突越严重

装填因子越小：

- 冲突越轻
- 但空间浪费更多

因此散列表的设计总是在做一个平衡：

> **拿更多空间，换更少冲突。**

---

## 5.3 完美散列只是特殊情形

课件第 17 页提到 perfect hashing：fileciteturn11file0

- 当数据集已知且固定时
- 可以用两级散列设计出完美散列
- 只需 `O(n)` 空间
- 最坏情况查找也可 `O(1)`

但课件也明确说：

> 在一般情况下，完美散列可期不可求。

因此本章主要关心的仍然是：

> **一般动态数据集下，如何降低冲突概率，并在冲突发生时尽快排解。**

---

## 5.4 生日悖论：冲突比想象中更早出现

课件第 18 页用生日悖论说明冲突的“高概率”：fileciteturn11file0

若把生日（月/日）散列到 365 个桶中，则：

- 23 个人时，至少两人生日相同的概率就已超过 50%

这说明：

> 即使桶数看起来挺大，只要装填因子略有上升，冲突就会很早变得普遍。

所以散列设计不能只是“碰运气”，必须认真处理：

1. 散列函数设计
2. 冲突排解策略

课件第 19 页也正是把本章任务概括为这两项。fileciteturn11file0

---

## 6. 散列函数：评价标准与设计原则

课件第 21 页给出四条标准：fileciteturn11file0

### 6.1 确定性（determinism）

同一关键码必须总被映射到同一地址。  
否则根本无法查找。

### 6.2 快速（efficiency）

计算 `hash(key)` 本身应该很快，理想是 expected `O(1)`。

### 6.3 满射（surjection）

尽可能充分覆盖整个散列空间，不要有大量桶永远用不到。

### 6.4 均匀（uniformity）

关键码映射到各桶的概率尽量接近，从而避免 clustering（数据聚集）。

可以把这四条总结成：

> **稳定、便宜、覆盖广、尽量均匀。**

## 7. 除余法与 MAD 法

## 7.1 最朴素的除余法

课件第 13 页给出最经典示例：fileciteturn11file0

\[
hash(key) = key mod M
\]

例如表长：

\[
M = 90001
\]

除余法优点是：

- 极其简单
- 计算快

---

## 7.2 为什么常常建议表长取素数

课件第 22 页讨论了一个常见经验：

- `M` 取素数时，分布往往更均匀

但课件也提醒：

> 对理想随机序列来说，素不素其实无关紧要；  
> 真正的问题在于**现实数据远非理想随机**。 fileciteturn11file0

现实 key 往往带有：

- 周期性
- 算术相关性
- 结构重复
- 语言统计规律

这时素数表长通常更不容易与这些规律“共振”。

---

## 7.3 除余法的两个缺陷

课件第 23 页指出：fileciteturn11file0

1. **不动点**  
   无论表长怎么取，总有：

\[
hash(0) = 0
\]

2. **相关性**  
   若关键码本身相邻，则散列地址也往往相邻

例如对连续整数做 `% M`，  
虽然整体上平均分散到了各桶，但相邻 key 仍映到相邻位置。  
这很容易导致 clustering。

---

## 7.4 MAD 法

为减弱这种简单相关性，课件给出：

> Multiply - Add - Divide（MAD）法 fileciteturn11file0

形式为：

\[
hash(key) = (a 	imes key + b) mod M
\]

其中：

- `M` 通常取素数
- `a > 1`
- `b > 0`
- 且 `M ∤ a`

它相当于先做一次线性变换，再取模。  
这样连续 key 的分布就不再机械地连续映射到连续桶。

---

## 8. 更多散列函数：构造整数地址

课件第 24～25 页列出了一系列传统方法：fileciteturn11file0

### 8.1 数字分析（selecting digits）

从 key 的十进制表示中抽取某几位构成地址。  
例如取奇数位。

适合那些字段结构已知、某些位更有区分力的场景。

### 8.2 平方取中（mid-square）

取：

\[
key^2
\]

的中间若干位作为地址。  
直觉上，它能把原始 key 中的高位 / 低位信息混合起来。

### 8.3 折叠法（folding）

把 key 分成若干段后求和，可能还带折返。

### 8.4 按位异或（XOR）

把 key 切成若干段，逐段异或，生成地址。

这些方法的共通目标都只有一个：

> **尽量打散原始 key 中可能存在的规律。**

---

## 9. 从字符串 / 对象到整数：`hashCode`

## 9.1 为什么字符串不能简单求和

课件第 27～28 页讨论字符串散列：fileciteturn11file0

例如不能只把字符编码简单求和，因为：

- 字符顺序信息会丢失
- 很多不同字符串会碰巧得相同和值

例如类似：

- `"abc"`
- `"cba"`

若只求字符和，就完全区分不开。

---

## 9.2 一个典型的字符串散列函数

课件第 27 页给出代码：fileciteturn11file0

```cpp
static size_t hashCode(char s[]) {
    unsigned int h = 0;
    for (size_t n = strlen(s), i = 0; i < n; i++) {
        h = (h << 5) | (h >> 27);
        h += (int) s[i];
    }
    return (size_t) h;
}
```

它做了两件事：

1. 对已有 hash 值做循环移位（类似混洗）
2. 再加上当前字符值

于是：

- 字符顺序被编码进去了
- 早期字符对最终结果仍保留影响
- 不会像简单求和那样严重丢信息

## 10. 随机数法与可移植性问题

课件第 30～31 页还提到一种思路：fileciteturn11file0

- 借助伪随机数发生器，把 key 映射为看似随机的整数

但课件很谨慎地指出：

> 把难题推给随机数发生器，并不一定好。  
> 因为不同平台、不同历史版本对 `rand()` 的实现可能不同，导致可移植性差。

因此这类方法必须慎用。

## 11. 冲突排解：开放散列（Open Hashing）

课件第 33～36 页先讲开放散列，也叫**分离链接法（separate chaining）**。fileciteturn11file0

### 11.1 多槽位（Multiple Slots）

最简单的想法是：

- 每个桶细分若干槽位
- 把同一个桶里的冲突词条都塞进去

问题在于：

- 很难预知到底要开多少槽
- 槽太多浪费空间
- 槽太少极端情况下仍不够

所以这只是直观起点。

---

### 11.2 独立链（Linked-List Chaining）

更成熟的做法是：

- 每个桶挂一条链表
- 所有映到该桶的同义词都放进链表里

优点：

1. 冲突次数再多也能容纳
2. 删除操作自然
3. 结构逻辑清晰

缺点：

1. 需要额外指针
2. 动态分配与回收有成本
3. 更重要的是：空间不连续，cache 不友好

课件第 35 页尤其强调了最后一点。fileciteturn11file0

---

### 11.3 公共溢出区（Overflow Area）

课件第 36 页还讲了另一种思路：fileciteturn11file0

- 所有冲突项统一顺序放到一块“公共溢出区”

优点：

- 实现简单

缺点：

- 一旦冲突发生，后续在溢出区中查找会退化到线性时间
- 最坏情况很差

---

## 12. 冲突排解：封闭散列（Closed Hashing / Open Addressing）

从第 37 页开始，课件转入更核心的内容：**开放定址**。fileciteturn11file0

这里术语上容易混淆：

- 课件把“Closed Hashing”与“Open Addressing”对应起来
- 意思是：冲突仍在散列表内部解决，而不是挂外链

---

### 12.1 Probe Sequence / Probe Chain

开放定址的关键概念是：

> 为每个 key 预先约定一条**试探链**（probe sequence）

查找时：

1. 先看 `hash(key)` 所在桶
2. 若冲突，则按试探链依次试后续桶
3. 直到：
   - 命中成功，或
   - 遇到真正空桶，判定失败

所以开放定址的本质问题变成：

> **试探链应当如何设计？**

---

## 13. 线性试探（Linear Probing）

## 13.1 定义

课件第 39 页给出：fileciteturn11file0

若 `hash(key)` 位置冲突，则依次尝试：

\[
h+1,\ h+2,\ h+3,\ ...
\pmod M
\]

即每次向后挪一个桶。

---

### 13.2 优点

课件总结了几个优点：fileciteturn11file0

1. 结构整体简洁，不需要链表或溢出区
2. 只要还有空桶，迟早能找到
3. 试探链连续，缓存局部性好

---

### 13.3 缺点：clustering

但线性试探最大的缺点是：

> **数据聚集（clustering）**

因为一旦有一段连续区间被占用，后续新词条就更容易继续堆到这段区间尾部，形成更长的连续块。  
这会进一步加剧冲突，形成恶性循环。

---

### 13.4 插入为什么容易，删除为什么难

课件第 40 页指出：fileciteturn11file0

- 插入：只需把新词条放到试探终止处的空桶里
- 删除：若简单把桶清空，会把某些试探链直接截断

这样后面原本存在的词条就可能“访问不到”了。

这说明开放定址中的删除，比插入微妙得多。

---

## 14. 懒惰删除（Lazy Removal）

## 14.1 基本思想

课件第 42 页给出解决方案：fileciteturn11file0

- 不真正清除为“全空”
- 而是额外用 `Bitmap removed` 给该桶打删除标记

于是被删桶在不同操作中的语义不同：

### 查找时

把它视作：

- 一个“不匹配的非空桶”

也就是：不能在这里停，必须继续沿试探链往后找。

### 插入时

把它视作：

- 一个可复用的空闲桶

于是新词条可以填进去。

---

### 14.2 代码逻辑

课件第 43 页给出两个核心函数：fileciteturn11file0

#### `probe4Hit(k)`

用于查找命中位置。  
遇到：

- 真冲突桶
- 懒删除标记桶

都要继续往后试探。

#### `probe4Free(k)`

用于找可插入位置。  
线性试探直到真正空桶。

这里体现出一个微妙区别：

- 懒删除标记桶对“查找”不是结束点
- 但对“插入”可以被回收利用

---

## 15. 平方试探（Quadratic Probing）

## 15.1 定义

课件第 45 页给出：fileciteturn11file0

若冲突，则依次尝试：

\[
h + 1^2,\ h + 2^2,\ h + 3^2,\ ...
\pmod M
\]

而不是线性递增。

---

### 15.2 优点

这样做的好处是：

- 试探步长越来越大
- 能更快地“跳离是非之地”
- clustering 现象有所缓解

相比线性试探，它不再机械地扫连续区间。

---

### 15.3 但并非总能找到空桶

课件第 46～47 页强调一个非常关键的问题：fileciteturn11file0

平方试探并不会像线性试探那样“只要有空桶就一定能找到”。

原因是：

- 平方模 `M` 的剩余类未必覆盖整个表
- 若 `M` 取值不合适，试探链可能只访问很少一部分桶

课件给出结论：

- 若 `M` 是素数，并且装填因子足够小
- 才能保证试探链前缀足够长，从而找到空桶

因此平方试探对：

- 表长
- 装填因子

都更敏感。

---

## 16. 双向平方试探

课件第 48～52 页继续改进平方试探：fileciteturn11file0

### 16.1 基本想法

不仅尝试：

\[
h + i^2
\]

还交替尝试：

\[
h - i^2
\]

即在两个方向上按平方距离扩张。

这样做的动机是：

> 让原本只覆盖“一半桶”的平方剩余类，有机会借助正反两个方向覆盖更多位置。

---

### 16.2 与数论的关系

课件第 51～52 页甚至引入了：

- `4k+3` 型素数
- 费马两平方和定理

来分析不同表长下，这种试探能否充分覆盖桶空间。fileciteturn11file0

这部分说明了一件很有趣的事：

> 开放定址中的试探链设计，本质上不只是编程技巧，还和模运算、数论性质密切相关。

---

## 17. 再散列（Double Hashing）

## 17.1 定义

课件第 54 页提出 double hashing：fileciteturn11file0

预先设计第二散列函数：

\[
hash_2(key)
\]

一旦冲突，之后的试探位置变为：

\[
hash(key) + i \cdot hash_2(key)
\pmod M
\]

---

## 17.2 为什么它更灵活

线性试探其实是 `hash2(key)=1` 的特殊情形。  
而 double hashing 的优点在于：

- 不同 key 会有不同试探步长
- 因而试探链重叠更少
- clustering 通常更轻

因此它常被认为是开放定址中性能较优的通用方案之一。

---

## 18. 重散列（Rehashing）

## 18.1 为什么必须重散列

课件第 56～58 页明确指出：fileciteturn11file0

随着装填因子上升：

- 冲突概率会激增
- 查找、插入、删除的平均代价都会恶化

因此当负载太高时，最有效的办法往往不是继续挣扎，而是：

> **整体搬迁到一张更大的新表。**

这就是 rehashing。

---

### 18.2 `rehash()` 的逻辑

课件第 56 页代码的要点是：fileciteturn11file0

1. 新建一张更大的表，表长取 `primeNLT(4*N)`
2. 清空所有懒删除标记
3. 扫描旧表
4. 把所有还存在的词条重新 `put()` 进新表

这样做的好处是：

- 装填因子重新降下来
- 旧的 clustering 被打散
- 懒惰删除留下的墓碑（tombstone）也被统一清理掉

---

### 18.3 插入与删除中的触发条件

课件第 57～58 页给出两个经验触发条件：fileciteturn11file0

#### 插入后

若：

\[
(N + L)	imes 2 > M
\]

即有效元素加懒删除标记太多，超过表长一半，则重散列。

#### 删除后

若：

\[
3N < L
\]

说明懒删除标记太多，墓碑已经严重干扰查找，也应重散列。

这些规则的本质都是：

> **把装填因子和墓碑比例控制在合理范围内。**

---

## 19. 桶排序（Bucket Sort）

从第 59 页开始，课件转入“散列思想还能做什么”。fileciteturn11file0

### 19.1 最简单情形：互异整数排序

若有 `n(<m)` 个取自 `[0,m)` 的互异整数，则可以开数组 / 散列表：

\[
H[0..m-1]
\]

然后：

1. 初始化全 0
2. 对每个输入 `key`，令 `H[key]=1`
3. 再顺序扫描 `H`，输出所有为 1 的位置

这样排序时间：

\[
O(n + m)
\]

若 `m = O(n)`，则是线性时间。

---

### 19.2 一般情形：允许重复

课件第 61 页指出：fileciteturn11file0

若允许重复，且可能 `m << n`，则每个桶中可挂一条链表存所有同义词。  
于是：

- 空间：`O(m+n)`
- 时间：`O(n)`

这说明散列 / 分桶思想不仅能支持查找，也能支持一些**非比较排序**。

---

## 20. MaxGap：利用分桶突破 `Ω(n log n)` 排序下界

课件第 62～65 页讨论一个很经典的问题：fileciteturn11file0

> 给定 `n` 个互异点，排序后相邻点间的最大间距是什么？

若直接排序后再扫一遍，需要：

\[
\Omega(n\log n)
\]

但课件给出一个线性算法。

---

### 20.1 核心思路

1. 找到全局最小值 `lo` 和最大值 `hi`
2. 将 `[lo, hi]` 均匀划成 `n-1` 段（`n` 个桶）
3. 每个桶只记录：
   - 桶内最小值
   - 桶内最大值
4. 最后只需看**相邻非空桶之间**的距离

---

### 20.2 为什么正确

课件第 65 页指出关键结论：fileciteturn11file0

> MaxGap 不可能局限在某个单独桶内，必然跨越至少两个桶。

原因是：

- 桶宽是按整体范围均匀分的
- 若两个点落在同一桶内，它们间距一定小于桶宽
- 真正最大的间距只能出现在相邻非空桶之间

于是根本不必知道桶内所有元素的完全有序关系。  
只记录每桶 `min / max` 即可。

这正体现了一个很深刻的算法思想：

> **不是所有排序相关问题都真的需要“完整排序”。**

---

## 21. 基数排序（Radix Sort）

## 21.1 词典序视角

课件第 67 页把多关键字段的排序看成“单词按词典序排序”：fileciteturn11file0

若 key 由多个域：

\[
k_d, k_{d-1}, ..., k_1
\]

组成，那么可以把它理解成一个 `d` 位“单词”。

---

### 21.2 LSD Radix Sort

课件第 68 页给出算法：

> 自最低位 `k_1` 到最高位 `k_d`，依次对每一位做一趟稳定桶排序。 fileciteturn11file0

---

### 21.3 为什么正确

课件第 69 页给出归纳证明思路：fileciteturn11file0

假设前 `i-1` 趟后，所有词条已经按低 `i-1` 位有序。  
第 `i` 趟后：

- 若第 `i` 位不同，则本趟桶排序会把它们排正
- 若第 `i` 位相同，则由于桶排序稳定，它们原本在低 `i-1` 位上的相对次序不变

于是归纳成立。

这说明基数排序的关键不是“桶排序本身”，而是：

> **每一趟必须稳定。**

---

### 21.4 复杂度

课件第 70 页写出总成本：fileciteturn11file0

\[
Oig(d\cdot(n+m)ig)
\]

其中：

- `d`：位数 / 域数
- `m`：单个域的取值范围上界

若：

- `m = O(n)`
- 且 `d` 可视为常数

则总复杂度可达线性。

---

## 22. 常对数密度整数集的线性排序

课件第 74～75 页进一步讲：fileciteturn11file0

若整数取值范围满足某种“常对数密度”条件，则可以：

1. 先转成 `n` 进制表示
2. 每个整数只含常数个 digit
3. 再用 radix sort 排序

从而得到：

\[
O(n)
\]

时间的整数排序算法。

课件特别强调，这之所以能“突破”比较排序下界，是因为：

- 这里输入是受限整数
- 计算模型不再是纯比较模型

---

## 23. 计数排序（Counting Sort）

课件第 76～84 页专门讲了 counting sort。fileciteturn11file0

### 23.1 算法思想

假设 key 取值属于一个较小集合，例如扑克牌花色。

步骤是：

1. 统计每个 key 出现次数
2. 对计数做前缀累加，得到每类 key 在结果数组中的秩区间
3. 自后向前扫描原数组
4. 将元素稳定地放到对应位置

---

### 23.2 为什么要从后往前扫

课件第 84 页还专门问：

> 最后一步能否改成从前向后？ fileciteturn11file0

答案是若想保持稳定性，通常要从后向前放。  
因为：

- 同一 key 的后出现元素，应放在更靠后的剩余位置
- 从后往前扫能自然保持相对顺序

所以 counting sort 也是一个**稳定排序**。

---

## 24. 跳转表（Skip List）：概率平衡的有序词典

从第 85 页开始，课件进入最后一部分：跳转表。fileciteturn11file0

这部分与前面散列表不同，它不是按对象直接散列，而是：

> **保留关键码的顺序结构，但不靠严格旋转维护平衡。**

---

## 24.1 动机

课件第 86 页引用 William Pugh 1989 年的论文题目：fileciteturn11file0

> *Skip lists are a probabilistic alternative to balanced trees.*

这句话就是跳转表的最佳总结：

- 它是平衡树的概率替代品
- 插入和删除实现比平衡树简单
- 但依然能达到期望 `O(log n)` 查找

---

## 24.2 结构：分层列表 + 竖直塔

课件第 87 页给出跳转表的结构：fileciteturn11file0

- 底层到顶层共有 `S0, S1, ..., Sh`
- 每层是一个有序链表
- 节点在纵向用 `above / below` 连成塔
- 水平用 `prev / next` 串起来

可以把它理解成：

> **在最底层完整有序链表之上，逐层随机抽稀，形成越来越稀疏的高速索引层。**

---

## 24.3 为什么空间仍然是线性的

课件第 88～89 页分析了空间：fileciteturn11file0

若每一层都由下一层节点以概率 `1/2` 提升而来，则：

- 每个节点塔高服从几何分布
- 平均塔高是常数
- 所有层节点总数期望仍为 `O(n)`

所以跳转表虽然“复制了很多层”，但总体空间并没有爆炸，仍是线性的。

---

## 25. 跳转表查找

## 25.1 基本策略：由高到低、由粗到细

课件第 93 页用非常形象的话总结：fileciteturn11file0

> 减而治之：由粗到细 = 由高到低

查找时：

1. 从最高层最左端开始
2. 在当前层尽量向右走，直到再走就会超过目标
3. 然后往下一层
4. 重复直到最底层

这与“在高速公路上先大步接近，再下匝道细找”非常像。

---

### 25.2 `skipSearch()` 的实现

课件第 94 页给出代码：fileciteturn11file0

```cpp
while (true) {
    while (p->succ && (p->entry.key <= k)) p = p->succ;
    p = p->pred;
    if (p->pred && (k == p->entry.key)) return true;
    qlist = qlist->succ;
    if (!qlist->succ) return false;
    p = p->pred ? p->below : qlist->data->first();
}
```

核心逻辑就是：

- 每层先往右走到“刚好不超过目标的位置”
- 然后下降一层继续

最终无论成功还是失败，返回的 `p` 都是：

> **不大于 `k` 的最大者**

这和 BST 中 `search()` 失败后返回最接近位置的思想很相似。

---

### 25.3 查找为什么是期望 `O(log n)`

课件第 95～97 页把查找成本拆成：

1. **纵向跳转次数**
2. **横向跳转次数** fileciteturn11file0

#### 纵向

层高本身是 `O(log n)` 的概率量级。

#### 横向

在任一层内，连续横跳次数的期望是常数。

于是总查找时间期望为：

\[
O(\log n)
\]

这正是跳转表最漂亮的结论：

> **不需要 AVL / 红黑树那种复杂局部重构，也能得到期望对数性能。**

---

## 26. 跳转表插入与删除

## 26.1 插入：自底向上长塔

课件第 99～100 页给出插入逻辑：fileciteturn11file0

1. 先调用 `skipSearch()` 找到底层插入位置
2. 在底层插入一个新节点
3. 然后反复“掷硬币”：
   - 若为正，则在上一层再插一个副本
   - 若当前已到最顶层，则先创建新层
4. 直到硬币失败为止

于是每次插入的新 key 会形成一座随机高度的塔。

这也说明跳转表的平衡完全来自：

> **随机化（coin flipping）**

而不是严格的旋转规则。

---

## 26.2 删除：逐层拆塔

课件第 103～104 页给出删除逻辑：fileciteturn11file0

1. 先查找目标塔
2. 若不存在，则失败返回
3. 若存在，则从当前层开始一路向下：
   - 删除当前层节点
   - 转到下一层对应副本
4. 最后若最顶层已空，则把空层移除

因此删除其实就是插入的逆过程。

---

## 27. 词典这一章的真正主线

这一章内容看起来很杂：

- 散列表
- 各种散列函数
- 各种冲突排解
- 重散列
- 多个线性排序
- 跳转表

但其实背后只有两条主线。

### 27.1 第一条主线：不依赖大小比较，直接按对象访问

这对应：

- Dictionary ADT
- Hashtable
- Hash Function
- Collision Resolution

关键思想是：

> **既然 key 未必适合比较大小，那就把 key 映射成桶地址。**

### 27.2 第二条主线：若仍想保序，但不想维护复杂平衡树

这对应：

- Skip List

关键思想是：

> **用分层随机索引，取代严格平衡树。**

---

## 28. 本章主线总结

### 28.1 词典是比 BST 更通用的映射抽象

它只要求：

- 能判等
- 能根据 key 找 value

不一定要求 key 可比较大小。

### 28.2 散列表的核心不在“直接”，而在“直接得足够好”

散列访问并不是严格最坏 `O(1)`，而是依赖于：

- 散列函数设计
- 装填因子控制
- 冲突排解策略

### 28.3 冲突不可避免，所以重点是概率降低 + 事后排解

- 好散列函数尽量均匀分布
- 好冲突策略尽量减少 clustering
- rehash 在必要时整体重建秩序

### 28.4 非比较排序展示了“分桶 / 计数 / 基数”的威力

- 桶排序
- counting sort
- radix sort
- max gap

都说明：  
若输入类型和范围受限，就不必被 `Ω(n log n)` 比较下界束缚。

### 28.5 Skip List 是“概率版平衡搜索结构”

- 结构比 AVL / 红黑树简单
- 查找、插入、删除都期望 `O(log n)`
- 非常适合作为有序词典的工程实现替代方案

---

## 29. 一句话总结

> 这一章真正重要的，不只是记住散列表和跳转表的代码，而是理解：当关键码不适合比较时，就用散列把对象直接映到地址；当仍想保留有序查找能力、又不想维护复杂旋转规则时，就用随机分层的跳转表来近似平衡搜索树。
