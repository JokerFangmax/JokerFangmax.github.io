# 数据结构与算法笔记（十二）：优先级队列

> 本文基于课程第十二章“优先级队列”整理。内容主线包括：  
> 优先级队列（PQ）的抽象接口与应用动机、几种朴素实现的复杂度比较、完全二叉堆的结构与堆序性、插入与删除的上滤 / 下滤、Floyd 自下而上建堆、堆排序、锦标赛树（胜者树与败者树）、多叉堆、左式堆以及最后简要提到的优先级搜索树（Priority Search Tree）。  
> 这一章真正的核心思想可以概括为一句话：  
> **如果我们反复只关心“当前最大 / 最小者是谁”，就没有必要维护所有元素的全序关系，只需维护足够支持极值访问的偏序结构。**  

---

## 1. 为什么需要优先级队列

### 1.1 应用动机

课件开头就给出若干非常典型的应用场景：

- 离散事件模拟
- 操作系统中的任务调度 / 中断处理 / MRU 等机制
- 输入法中的词频调整
- 排序算法
- 贪心算法（如 Huffman 编码、Kruskal）
- 平面扫描算法中的事件队列 

这些场景有一个共同点：

> 我们总是需要**反复地、快速地定位当前极值元素**。

并且课件特别强调了两个动态性：

1. 集合组成是动态变化的  
   也就是元素会不断插入、删除
2. 元素优先级本身也可能动态变化 

所以问题不是“静态求一次最大值”，而是：

> 在一个不断变化的集合中，高效维护“当前最优者”。

---

### 1.2 PQ 的抽象接口

课件第 3 页给出 PQ 作为 ADT 的典型接口：

```cpp
template <typename T> struct PQ { // priority queue
    virtual void insert( T ) = 0;
    virtual T getMax() = 0;
    virtual T delMax() = 0;
};
```

这三个操作正是 PQ 的核心：

- `insert(e)`：插入一个元素
- `getMax()`：查看当前最大元
- `delMax()`：删除并返回当前最大元

注意，PQ 的重点不是“遍历全部元素”，而是：

> **极值访问（access by priority）**

---

### 1.3 栈、队列、Steap、Queap 都可看作 PQ 特例

课件第 3 页指出：

- Stack 和 Queue 都是 PQ 的特例，因为优先级完全由插入次序决定
- Steap 和 Queap 也是 PQ 的特例，只是插入、删除位置受限 

这说明：

> 优先级队列并不等于“堆”本身；  
> 堆只是实现 PQ 的一种高效方式。

从抽象层面看，PQ 是一种更宽泛的访问模型：

- 栈：后进先出，相当于“最新元素优先”
- 队列：先进先出，相当于“最老元素优先”
- 普通 PQ：优先级由关键码 / 权值决定

---

## 2. 朴素实现：为什么有序结构未必划算

课件第 5～9 页先系统比较了几种直接实现方式。  
这一部分非常重要，因为它解释了为什么我们最终会走向“堆”。

---

## 2.1 基于无序向量（Vector）

课件第 5 页给出的复杂度是：

- `getMax()`：遍历整个向量，`Θ(n)`  ***为什么用θ***
- `delMax()`：先遍历找最大，再删除，`Θ(n)`
- `insert()`：直接尾插，`O(1)`

这说明无序向量的特点是：

- 插入很快
- 但每次取极值都很慢

如果应用里：

- 插入很多
- 但极值访问很少

它也许还能接受。  
但 PQ 的典型使用场景偏偏是：

> “反复找极值”

所以这里会很吃亏。

---

## 2.2 基于有序向量（Sorted Vector）

课件第 6 页给出的结论是：

- `getMax()`：直接看末元素，`O(1)`
- `delMax()`：删除末元素，`O(1)`
- `insert()`：先搜索位置，再插入搬移，`O(log n) + O(n) = O(n)`

这与无序向量正好相反：

- 取极值很快
- 插入很慢

---

## 2.3 基于无序列表（List）

课件第 7 页给出的结论：

- `getMax()`：遍历找最大，`Θ(n)`
- `delMax()`：遍历后删除，`Θ(n)`
- `insert()`：表头插入，`O(1)`

本质上与无序向量非常接近，只是：

- 删除已知位置元素在链表里更方便
- 但“找最大”仍绕不过线性扫描

---

## 2.4 基于有序列表（Sorted List）

课件第 8 页给出：

- `getMax()`：直接看表头，`O(1)`
- `delMax()`：删除表头，`O(1)`
- `insert()`：需线性查找位置，`O(n)`

它与有序向量一样：

- 极值访问快
- 插入慢

---

## 2.5 基于 BBST

课件第 9 页指出：AVL、Splay、Red-Black 三种平衡搜索树都可以做到：

- `insert()`：`O(log n)`
- `getMax()` / `delMax()`：也可做到 `O(log n)` 或其中 `getMax()` 更优

但课件紧接着提出一个非常关键的判断：

> BBST 的功能远远超出了 PQ 的需求。 

因为 BBST 维护的是：

- 全体元素之间的**全序关系**

而优先级队列通常只关心：

- 当前极值
- 以及插入 / 删除后如何维持“足够找出极值”的局部秩序

所以课件总结：

> 若只需查找极值元，则不必维护所有元素之间的全序关系，偏序足矣。 

这句话就是“堆”诞生的核心动机。

---

## 3. 为什么“偏序”就够了

### 3.1 全序 vs 偏序

在有序向量 / 有序列表 / BBST 中，我们维护的是：

> 任意两个元素之间都能比较并确定先后

这叫**全序**。

但如果目标只是：

- 反复找到最大元
- 删除最大元
- 插入新元素后恢复这种能力

其实并不需要知道任意两个非极值元素之间谁大谁小。  
只要保证：

- 某些局部父子关系成立
- 从而全局极值一定浮到某个固定位置

就够了。

这就是：

> **堆序性（heap order）**  
> 一种比全序弱得多、但对 PQ 已经足够的偏序关系。

---

### 3.2 为什么这会带来效率优势

若只维持偏序而非全序：

- 更新成本更低
- 调整范围更局部
- 实现也更简单
- 常数因子往往更小

课件第 9 页正是基于这一点提出：

> 应该存在某种更简单、维护成本更低的实现方式，  
> 仍能把各接口做成 `O(log n)`，而且实际效率更高。 

这个“更简单的实现方式”，就是**堆（heap）**。

---

## 4. 完全二叉堆：结构与堆序性

从课件第 11 页开始，进入本章核心：**完全二叉堆（complete binary heap）**。

---

## 4.1 为什么选“完全二叉树 + 向量”

课件第 12 页强调完全二叉堆有两个层面：

- **逻辑上**：等同于一棵完全二叉树
- **物理上**：直接借助向量实现

这是一种非常漂亮的结合：

1. 完全二叉树保证了高度平衡  
   树高始终是 `O(log n)`
2. 层次遍历次序与数组下标天然对应  
   因而无需显式指针

---

## 4.2 下标关系

课件第 12 页给出三个经典宏：

```cpp
#define Parent(i) ( ((i) - 1) >> 1 )
#define LChild(i) ( 1 + ((i) << 1) )
#define RChild(i) ( (1 + (i)) << 1 )
```

也就是：

- 父亲下标：

$
Parent(i)=\lfloor (i-1)/2 
floor
$

- 左孩子下标：

$
LChild(i)=2i+1
$

- 右孩子下标：

$
RChild(i)=2i+2
$

这说明完全二叉堆之所以工程上极其方便，关键就在于：

> **结构信息可由秩直接计算，而无需指针。**

---

## 4.3 什么是堆序性

课件第 14 页给出最大堆的堆序性：

> 只要 `0 < i`，就必满足
$
H[i] \le H[Parent(i)]
$

也就是说：

- 每个节点都不大于其父亲
- 因而整棵树根节点一定是全局最大

于是 `getMax()` 只需返回数组第 0 个元素。  
课件直接给出：

```cpp
template <typename T> T PQ_ComplHeap<T>::getMax() { return _elem[0]; }
```

---

## 4.4 完全堆的两个基本性质

因此完全二叉堆同时满足：

### 结构性

它是一棵完全二叉树，所以高度是：

$
O(\log n)
$

### 堆序性

它只要求：

- 父亲不小于孩子

而不要求不同子树之间整体有序。

正因为只维护这种偏序，堆才能做到：

- 极值访问快
- 插入 / 删除时只需沿一条根叶路径局部调整

---

## 5. 完全二叉堆的插入：上滤

## 5.1 插入思路

课件第 16～21 页用连环图展示了插入过程：

1. 新元素先放到向量末尾  
   这对应完全二叉树的最后一个叶位
2. 若它比父节点更大，就与父交换
3. 重复这一过程，直到：
   - 到达根，或
   - 父子已经满足堆序

这就叫：

> **逐层上滤（percolate up）**

---

## 5.2 插入实现

课件第 22 页给出代码：

```cpp
template <typename T> void PQ_ComplHeap<T>::insert( T e ) {
    Vector<T>::insert( e );
    percolateUp( _elem, _size - 1 );
}
```

其中：

```cpp
template <typename T> Rank percolateUp( T* A, Rank i ) {
    while ( 0 < i ) {
        Rank j = Parent( i );
        if ( lt( A[i], A[j] ) ) break;
        swap( A[i], A[j] );
        i = j;
    }
    return i;
}
```

它完整体现了上滤逻辑：

- 比父大 → 交换
- 换完继续看更高层
- 一旦不再违反堆序 → 停止

---

## 5.3 为什么插入复杂度是 `O(log n)`

课件第 23 页给出理由：

- 新元素上滤时，只可能与祖先交换
- 完全二叉树的祖先数不超过树高
- 而树高是 `O(log n)`

因此：

$
insert = O(\log n)
$

不过课件也提醒：

> 数学期望意义下，实际效率往往远远更高。 

因为随机数据下，通常不会真的一路上滤到根。

---

## 6. 完全二叉堆的删除：下滤

## 6.1 删除最大元的思路

课件第 25～30 页用“割肉补疮 + 逐层下滤”来形容删除：

删除堆顶时不能直接把根拿掉后留空，因此标准做法是：

1. 备份根元素（即最大元）
2. 用最后一个叶节点填补堆顶
3. 堆规模减一
4. 再把新的根节点向下调整到合适位置

这个过程叫：

> **下滤（percolate down）**

---

## 6.2 删除实现

课件第 31 页给出：

```cpp
template <typename T> T PQ_ComplHeap<T>::delMax() {
    T maxElem = _elem[0];
    _elem[0] = _elem[ --_size ];
    percolateDown( _elem, _size, 0 );
    return maxElem;
}
```

以及：

```cpp
template <typename T> Rank percolateDown( T* A, Rank n, Rank i ) {
    Rank j;
    while ( i != ( j = ProperParent( A, n, i ) ) ) {
        swap( A[i], A[j] );
        i = j;
    }
    return i;
}
```

这里 `ProperParent(A,n,i)` 的意思是：

- 在 `i` 及其孩子中，找出“最适合做父节点”的那个位置
- 对于最大堆，就是三者中的最大者

若最大者不是 `i` 自己，则交换并继续往下。

---

## 6.3 为什么删除复杂度也是 `O(log n)`

课件第 32 页总结：

- 新堆顶元素在每一层至多交换一次
- 可下移的层数不超过堆高
- 因而总交换次数不超过 `O(log n)`

所以：

$
delMax = O(\log n)
$

这与 `insert` 同阶。

---

## 7. 批量建堆：为什么 Floyd 算法能做到 `O(n)`

## 7.1 朴素建堆：逐个插入

课件第 34～36 页先给出一种直观做法：

若数组已有 `n` 个元素，希望把它们变成堆，可按层次遍历顺序逐一做 `percolateUp`：

```cpp
for ( Rank i = 1; i < n; i++ )
    percolateUp( A, i );
```

但这相当于：

- 第 1 个元素插入
- 第 2 个元素插入
- …
- 第 n 个元素插入

总复杂度最坏可达：

$
O(n \log n)
$

课件甚至指出：这么长的时间，本足以全排序，没办法体现堆的优势

---

## 7.2 Floyd 自下而上建堆

课件第 37 页给出 Robert Floyd 1964 的经典算法：

```cpp
template <typename T>
void heapify( T* A, Rank n ) {
    for ( Rank i = n/2 - 1; 0 <= i; i-- )
        percolateDown( A, n, i );
}
```

也就是说：

- 从最后一个内部节点开始
- 依次对所有内部节点做下滤
- 顺序是自底向上、从右到左

这可以理解为：

> **把许多小子堆不断合并成更大的合法堆。**

---

## 7.3 为什么是 `O(n)` 而不是 `O(n log n)`

课件第 39 页给出关键直觉：

- 不同内部节点下滤的成本，并不都达到 `log n`
- 越靠近底层的节点，高度越小，所需下滤步数越少
- 只有极少数靠近根的节点下滤成本大

因此总成本应按“高度总和”来算，而不是简单粗暴地乘上 `n log n`。

更形式化地，若看满二叉树：

- 高度为 `k` 的节点大约有 `2^{h-k}` 个
- 每个节点下滤成本与其高度 `k` 成正比
- 累加后得到线性级别

于是：

$
heapify = O(n)
$

这正是优先级队列 / 堆排序中非常关键的一个结论。

>为什么换了个建堆方向(由自上而下建堆变为自下而上建堆)会导致运算复杂度的变换？
- 自上而下建堆中有n/2个点需要上移O(log n)步，所以是O(n log n)
- 而自下而上建堆中数量更多的叶子节点需要移动的步数变少了(从深度变成了高度，即h-深度)，所以累积复杂度少了很多，变为O(n)

---

## 8. 堆排序（Heap Sort）

从课件第 41 页开始进入堆排序。

## 8.1 堆排序的核心不变性

课件第 42 页指出：

在 `selectionSort()` 中，若把无序区 `U` 改造成堆 `H`，则：

- 初始化：`heapify()`，`O(n)`
- 迭代：反复 `delMax()`，每次 `O(log n)`
- 不变性：堆区 `H` 中所有元素都不大于已排序区 `S`

因此总时间：

$
O(n) + n \times O(\log n) = O(n \log n)
$

这就是堆排序。

---

## 8.2 就地堆排序

课件第 43～44 页进一步指出：

完全二叉堆本身就是一个向量区间，  
因此无需额外开第二个数组来存已排序区。

做法是：

1. 先在区间 `[lo, hi)` 上建堆
2. 反复交换：
   - 堆顶 `A[0]`
   - 当前区间末元素 `A[n-1]`
3. 然后把堆规模减一，再对新堆顶下滤

代码如下：

```cpp
template <typename T>
void Vector<T>::heapSort( Rank lo, Rank hi ) {
    T* A = _elem + lo; Rank n = hi - lo;
    heapify( A, n );
    while ( 0 < --n ) {
        swap( A[0], A[n] );
        percolateDown( A, n, 0 );
    }
}
```

这正是经典的**原地堆排序**。

---

## 8.3 堆排序的特点

堆排序有几个很重要的性质：

### 时间复杂度

$
O(n \log n)
$

### 额外空间

$
O(1)
$

因为是原地排序。

### 稳定性

一般**不稳定**。  
因为上滤、下滤与交换会打乱相同关键码元素的相对次序。

因此堆排序在“稳定性要求不高、且希望严格原地”的场景中非常有吸引力。

---

## 9. 锦标赛树：胜者树与败者树

从课件第 48 页开始，转入另一类 PQ 实现：**锦标赛树（Tournament Tree）**。

## 9.1 胜者树（Winner Tree）

课件第 49 页给出定义：

- 叶节点：待比较元素（选手）
- 内部节点：孩子中的胜者
- 根节点：全局冠军

这本质上是把“找最大元”看成一场逐层淘汰赛。

---

## 9.2 胜者树的操作复杂度

课件第 49 页给出：

- `create()`：`O(n)`
- `remove()`：`O(log n)`
- `insert()`：`O(log n)` 

原因很简单：

- 构造时自底向上比赛一次即可
- 删除冠军后，只需沿原冠军叶节点到根这条路径重新比赛
- 路径长度正比于树高

---

## 9.3 用于排序

课件第 50～53 页展示了用胜者树做排序：

1. 建胜者树
2. 反复删除根（当前冠军）
3. 沿其叶路径回溯并重赛
4. 直到所有叶子都被“失活”

总复杂度同样是：

$
O(n \log n)
$

---

## 9.4 与堆的比较

课件第 54 页指出：

若只想从 `n` 个元素中找最小的 `k` 个（其中 `k \ll n`）：

- 胜者树与小顶堆在渐近意义上相当
- 但常数因子不同

尤其课件强调：

- 在堆的 `percolateDown()` 中，每层通常要做 2 次比较
- 而胜者树重赛路径上的比较组织方式不同，常数可能更优

这表明：

> 渐近复杂度相同，并不意味着工程效率完全一致。

---

## 9.5 败者树（Loser Tree）

课件第 55～58 页进一步提出败者树。

其思路是：

- 内部节点不再存胜者，而是存该轮比赛的**败者**
- 额外在根之上加一个“父节点”存冠军

这样做的好处是：

> 重赛时不必来回访问兄弟，只需沿着冠军路径直接回放比赛结果。

败者树在外部排序、多路归并中尤其经典，因为：

- 频繁取出当前最小者
- 再让同一路的新元素补位
- 重赛路径非常自然

---

## 10. 多叉堆（d-heap）

从课件第 59 页开始，讨论完全二叉堆的推广：**多叉堆**。

## 10.1 为什么要考虑 d-heap

课件第 60～61 页是从图的 PFS 算法切入的：

若在 Dijkstra / Prim 中用优先级队列管理顶点，则需要频繁做：

- `heapify()`：建初始 PQ
- `delMax()` 或对偶的 `delMin()`
- `increase()` / `decrease()`：更新顶点优先级

二叉堆能把这些做成：

- `heapify()`：`O(n)`
- `delMax()`：`O(log n)`
- `increase()`：`O(log n)`

但课件追问：

> 能否进一步优化？

---

## 10.2 d-heap 的平衡：上山容易下山难

课件第 62～63 页给出核心判断：

若把二叉堆改成 d 叉堆，则：

- 树高下降到：

$
O(\log_d n)
$

因此：

- **上滤**成本会下降

但与此同时：

- 每个节点孩子变多
- **下滤**时寻找“最合适孩子”的成本会上升

课件用一句非常形象的话总结：

> **上山容易下山难** 

也就是说：

- `increase()` / `percolateUp()` 会更快
- `delMax()` / `percolateDown()` 则可能更慢

---

## 10.3 为什么它对 PFS 有意义

在图算法中，PQ 的两种操作出现频率往往不一样：

- `delMin()` / `delMax()`：共 `n` 次
- `increase()` / `decrease()`：共 `e` 次

若图较稀疏，`e` 仍可能远大于 `n`。  
因此若能显著降低“上滤型更新”的代价，即使“下滤型删除”略变慢，也可能整体更优。

课件第 64 页正是说明：

- 通过合适选取 `d`
- 可以使 PFS 总体性能达到更好的平衡 

这说明 d-heap 不是单纯的堆推广，而是：

> **针对特定操作频率分布做性能权衡。**

---

## 11. Fibonacci 堆：理论上更强的优先级队列

课件第 65 页只做了简要提及：

- 借助“新的上滤算法 + 懒惰合并”
- 可以得到 Fibonacci 堆
- 其分摊复杂度为：
  - `delMax()`：`O(log n)`
  - `insert()`：`O(1)`
  - `merge()`：`O(1)`
  - `increase()`：`O(1)`

这正是很多教科书里用于优化：

- Dijkstra
- Prim
- 一般 PFS

的理论最优数据结构。

不过本套课件并未展开 Fibonacci 堆细节，而是转向一种更容易讲清楚的“可合并堆”——左式堆。

---

## 12. 左式堆：为什么要支持 merge

从课件第 66 页开始进入左式堆（Leftist Heap）。

## 12.1 问题：两堆合并能否高效完成？

课件第 67 页先问一个非常自然的问题：

给定两个堆 `A` 和 `B`，要构造：

$
H = merge(A, B)
$

若不用专门设计，可能有两种朴素办法：

### 方法一

不断做：

$
A.insert(B.delMax())
$

总时间约：

$
O(m \log (n+m))
$

### 方法二

把所有元素 union 后重新 heapify

总时间：

$
O(m+n)
$

课件接着追问：

> 能否更快？比如做到 `O(log n)` 量级？ 

普通完全二叉堆由于结构性太强，很难支持高效 `merge()`。  
于是需要一种：

- 堆序性仍然保留
- 但结构限制放松

的新堆结构。

---

## 13. 左式堆的核心思想：只控制右侧藤长

## 13.1 沿右侧藤合并

课件第 68～72 页通过图示说明：

两堆合并本质上像“两路归并”，  
但为了保持堆序，只需统一沿右侧分支（right spine / 右侧藤）递归下去。

于是：

- merge 的真实代价
- 不取决于整棵树高度
- 而取决于右侧藤总长

这就把问题转化成了：

> **如何让右侧藤始终足够短？**

---

## 13.2 NPL：Null Path Length

课件第 75 页引入定义：

对节点 `x`，`npl(x)` 表示：

> 从 `x` 出发到外部空节点的最近距离

也叫 Null Path Length。  
可以理解为：

- 离最近空位有多远
- 或等价地，以 `x` 为根的最大满子树高度

---

## 13.3 左式堆的定义

课件第 76 页指出：

左式堆在保持堆序性的同时，还额外要求：

$
npl(lc(x)) \ge npl(rc(x))
$

也就是说：

> 任一节点的左子堆都“不比右子堆更浅”

这意味着整棵树倾向于：

- 节点更多地堆在左边
- 右侧链尽量短

课件称之为：

> **左式堆 = 处处左倾** 

---

## 13.4 为什么右侧链长是 `O(log n)`

课件第 77 页给出结论：

若右侧链长度为 `d`，则左式堆至少包含：

$
2^d - 1
$

个内部节点（等价量级表达）。  
因此反过来，对包含 `n` 个节点的左式堆，其右侧链长度必为：

$
O(\log n)
$

这正是左式堆能高效 `merge()` 的理论根基。

---

## 14. 左式堆的 merge 算法

## 14.1 类定义

课件第 79 页给出框架：

```cpp
template <typename T>
class PQ_LeftHeap : public PQ<T>, public BinTree<T> {
public:
    T getMax() { return _root->data; }
    void insert(T);
    T delMax();
    PQ_LeftHeap( PQ_LeftHeap & A, PQ_LeftHeap & B ) {
        _root = merge(A._root, B._root);
        _size = A._size + B._size;
        A._root = B._root = NULL;
        A._size = B._size = 0;
    }
};
```

可见：

- `getMax()` 仍直接看根
- `insert()` 和 `delMax()` 都将基于统一的 `merge()` 实现

这也是左式堆最漂亮的地方：

> **merge 是核心原语，其余操作都可化归为 merge。**

---

## 14.2 merge 的递归结构

课件第 81 页给出核心代码：

```cpp
template <typename T>
static BinNodePosi<T> merge( BinNodePosi<T> a, BinNodePosi<T> b ) {
    if ( !a ) return b;
    if ( !b ) return a;
    if ( lt( a->data, b->data ) ) swap( b, a );
    ( a->rc = merge( a->rc, b ) )->parent = a;
    if ( !a->lc || a->lc->npl < a->rc->npl )
        swap( a->lc, a->rc );
    a->npl = a->rc ? 1 + a->rc->npl : 1;
    return a;
}
```

可以分成三步理解：

### 第一步：保证 `a` 的根更大

若 `b` 更大，则交换，使得：

- 合并后的堆根一定取两者中较大者

这保证了堆序性。

### 第二步：递归合并 `a->rc` 与 `b`

因为左式堆只关心控制右侧链，所以统一沿右子堆递归合并。

### 第三步：若必要则交换左右子树

递归回来后，若：

$
npl(lc) < npl(rc)
$

就交换左右子树，以恢复左倾性。

最后更新 `npl(a)`。

---

## 14.3 为什么 merge 是 `O(log n)`

因为递归只沿右侧链深入，  
而左式堆的右侧链长度始终是 `O(log n)`。  
因此：

$
merge = O(\log n)
$

这也是左式堆相比完全二叉堆最大的特色：

> **支持高效合并**

---

## 15. 左式堆的插入与删除

## 15.1 插入

课件第 88 页给出：

```cpp
template <typename T>
void PQ_LeftHeap<T>::insert( T e ) {
    _root = merge( _root, new BinNode<T>( e, NULL ) );
    _size++;
}
```

也就是说，插入一个元素等价于：

- 把它看成单节点堆
- 再与原堆做一次 merge

因此：

$
insert = O(\log n)
$

---

## 15.2 删除

课件第 89 页给出：

```cpp
template <typename T>
T PQ_LeftHeap<T>::delMax() {
    BinNodePosi<T> lHeap = _root->lc; if (lHeap) lHeap->parent = NULL;
    BinNodePosi<T> rHeap = _root->rc; if (rHeap) rHeap->parent = NULL;
    T e = _root->data;
    delete _root; _size--;
    _root = merge( lHeap, rHeap );
    return e;
}
```

这也非常自然：

1. 删除根
2. 剩下左、右两棵子堆
3. 再 merge 回去

因此：

$
delMax = O(\log n)
$

## 16. 优先级搜索树（PST）简述

课件最后第 90～95 页简要介绍了 **Priority Search Tree**。

虽然这一部分在本章中只是简略带过，但它很有代表性，因为它展示了：

> **Priority Search Tree = BST + PQ** 

也就是说，它同时融合了：

- BST 按 x 坐标组织的搜索性
- PQ 按 y 坐标维护极值的能力

课件第 91～95 页围绕一个二维查询问题（Grounded Range Query）来讲解：

- 对坏的 `x` 直接剪枝
- 对坏的 `y` 也可剪枝
- 接受的节点直接输出
- 查询时间最终为：

$
O(r + \log n)
$

其中 `r` 为输出规模。 

这其实是在提醒我们：

> 优先级队列思想并不只用于“调度”或“排序”，  
> 它也可以嵌入几何数据结构中，与搜索树结合。

## 17. 本章主线总结

### 17.1 优先级队列只关心极值，不关心全序

因此：

- 有序向量 / BBST 虽然能做 PQ
- 但往往维护成本过高
- 堆通过偏序关系恰好满足需求

### 17.2 完全二叉堆是最经典的 PQ 实现

它结合了：

- 完全二叉树的低高度
- 向量实现的紧凑结构
- 上滤 / 下滤的局部调整

从而支持：

- `getMax()`：`O(1)`
- `insert()`：`O(log n)`
- `delMax()`：`O(log n)`

### 17.3 Floyd 建堆是线性时间的关键技巧

批量建堆时：

- 逐个插入是 `O(n log n)`
- 自下而上下滤是 `O(n)`

这一点不仅影响 PQ，也直接影响堆排序的初始化阶段。

### 17.4 堆排序展示了 PQ 在排序中的作用

把“反复选最大”这个过程嵌入到原地数组上，就得到：

- 原地
- `O(n log n)`
- 不稳定

的堆排序。

### 17.5 胜者树 / 败者树适合反复选极值并局部回放

它们与堆在渐近复杂度上相近，但常数因子、重赛路径、实际工程场景可能不同，尤其在多路归并中很常见。

### 17.6 d-heap、左式堆、Fibonacci 堆分别优化不同操作模式

- **d-heap**：平衡上滤与下滤成本，适合某些图算法场景
- **左式堆**：核心在于高效 `merge`
- **Fibonacci 堆**：理论上给出更优的分摊复杂度

### 17.7 Priority Search Tree 说明 PQ 思想还能与搜索树融合

它提示我们：

> 优先级队列并不只是一个孤立结构，而是很多更复杂数据结构与算法的基础模块。

---

## 18. 一句话总结

> 这一章真正重要的，不只是学会完全二叉堆的上滤、下滤与堆排序，而是理解：当问题只要求我们反复定位当前极值时，维护全序是一种浪费；通过堆这样的偏序结构，我们可以用更低的维护成本，获得足够强且更高效的优先级访问能力。
