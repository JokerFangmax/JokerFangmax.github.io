# 数据结构与算法笔记（四）：栈与队列

> 本文基于课程第四章“栈与队列”整理。内容不只是按 PPT 顺序摘录，而是围绕“受限线性结构”的思想，把栈、调用栈、递归消除、进制转换、括号匹配、栈混洗、中缀表达式求值、逆波兰表达式、队列、双栈当队、Steap / Queap，以及直方图最大矩形等问题系统串起来，整理成适合博客发布的 Markdown 笔记。内容主要依据课件中的定义、代码与示意图展开。 fileciteturn6file0

---

## 1. 栈：受限的线性序列

### 1.1 什么是栈

栈（stack）是一种**受限的序列**：

- 只能在**栈顶**（top）插入
- 只能在**栈顶**删除
- 栈底（bottom）是盲端

课件第一页就把这个性质总结成：

- 后进先出（LIFO, Last In First Out）
- 或等价地，先进后出（FILO, First In Last Out） fileciteturn6file0

这说明栈并不是“新数据结构”，而是：

> **在线性序列的基础上，对可操作位置做了进一步限制。**

---

### 1.2 栈的基本接口

栈的接口非常小，但非常“纯”：

- `size()`：当前元素个数
- `empty()`：是否为空
- `push(e)`：入栈
- `pop()`：出栈
- `top()`：查看栈顶元素

课件还提到扩展接口，例如：

- `getMax()`：查询栈内最大值 fileciteturn6file0

这也说明：

> **栈的本质限制的是修改方式，而不是禁止做额外查询。**

---

### 1.3 一个简单例子

如果栈从底到顶依次是：

```text
5, 7, 3
```

那么：

- `top()` 返回 `3`
- `pop()` 删除 `3`
- 再 `push(11)` 后，栈变成 `5, 7, 11`

课件第 3 页给出了一串详细操作示例，核心就是让你看到：

- 栈顶是唯一可访问的“活动端”
- 所有动作都围绕这一端展开 fileciteturn6file0

---

## 2. 栈的实现：为什么可以直接基于向量或列表

### 2.1 栈是“序列的特例”

课件指出：

> 栈既然属于序列的特例，故可直接基于向量或列表派生。 fileciteturn6file0

这句话非常重要。  
因为这意味着：

- 栈本身不需要重新发明存储结构
- 只要在已有线性结构上**限制接口**即可

这和前面“ADT 与实现分离”的思想是一致的。

---

### 2.2 基于向量实现栈

课件代码如下：fileciteturn6file0

```cpp
template <typename T>
class Stack: public Vector<T> {
public:
    void push(T const& e) { insert(e); }          // 入栈
    T pop() { return remove(size() - 1); }        // 出栈
    T& top() { return (*this)[size() - 1]; }      // 查顶
};
```

这里默认：

- 向量**末端**作为栈顶
- 向量**首端**作为栈底

---

### 2.3 为什么这些操作都是 `O(1)`

若以向量末端作为栈顶：

- `push(e)` 相当于尾插
- `pop()` 相当于尾删
- `top()` 相当于读最后一个元素

在动态数组实现下：

- `top()` 是 `O(1)`
- `pop()` 是 `O(1)`
- `push()` 单次最坏可因扩容达 `O(n)`，但**分摊复杂度**为 `O(1)`

所以课程里会说：

> 如此实现的栈接口，均只需 `O(1)` 时间。 fileciteturn6file0

严格来说，这里的 `push()` 应理解为**分摊 `O(1)`**。

---

### 2.4 如果反过来把向量首端当栈顶呢

课件特意留了一个思考题：

> 以向量首 / 末端为栈底 / 顶 —— 颠倒过来呢？ fileciteturn6file0

答案是效率会明显变差。

若把向量首端当栈顶：

- `push(e)` 需要在首部插入，后缀整体后移
- `pop()` 需要删除首部元素，后缀整体前移

于是这两个操作都会退化为：

\[
O(n)
\]

这说明：

> **同样是“栈”，底层依赖的序列哪一端作为活动端，会直接决定效率。**

---

### 2.5 基于列表实现栈

若基于列表实现，思路同样简单：

- 选列表的首端或末端作为栈顶
- 只要插入和删除都发生在同一端即可

由于双向链表在已知位置时：

- 插入是 `O(1)`
- 删除是 `O(1)`

因此基于列表实现栈也可以把核心接口做到 `O(1)`。  
但工程中更常见的是基于动态数组，因为：

- 局部性更好
- 实现简单
- 常数更小

---

## 3. 调用栈：程序运行时为什么天然需要栈

第四章第二部分其实非常关键，它说明：

> 栈不仅是一种“程序员手写的数据结构”，更是函数调用机制的天然模型。 fileciteturn6file0

---

### 3.1 函数调用树如何落地为执行过程

课件用“函数调用树：如何实现？”这个问题引出调用栈。  
递归或嵌套函数调用在逻辑上看起来像一棵树，但实际执行时只能是线性的时间顺序。 fileciteturn6file0

那么程序到底如何记住：

- 目前执行到哪个函数
- 返回时应跳回哪里
- 每层调用有哪些局部变量

答案就是：

> **用调用栈（call stack）维护函数帧（stack frame）。**

---

### 3.2 每个调用帧里有什么

从课件第 7～10 页的图可以看出，一个函数帧通常至少包含： fileciteturn6file0

- 返回地址（return address）
- 前一帧位置（previous frame）
- 形参与局部变量
- 可能还有保存寄存器等上下文信息

例如：

- `main()` 调用 `funcA()`
- `funcA()` 再调用 `funcB()`
- `funcB()` 又递归调用 `funcB()`

那么调用栈就会一层层向上增长。  
最近的调用总在栈顶，这正好符合：

- 最后调用的函数最先返回
- 即典型的 LIFO 结构

---

### 3.3 调用栈与程序地址空间的关系

课件右侧“binary executable”与左侧“call stack”的配图非常形象：  
程序代码本身固定在可执行文件对应的地址区域，而运行过程中，函数每被调用一次，就在调用栈里生成一个新的活动记录。 fileciteturn6file0

这说明：

- 代码是“静态的”
- 调用帧是“动态的”
- 同一个函数可以同时有多个活动实例，只要它被递归或多次调用

---

## 4. 调用栈实例：递归到底怎么占空间

### 4.1 `fac(n)` 的调用栈

课件给出：

```cpp
int fac(int n) { return (n < 2) ? 1 : n * fac(n - 1); }
```

在调用 `fac(3)` 时，调用栈会大致经历：

1. `fac(3)` 入栈，等待 `fac(2)`
2. `fac(2)` 入栈，等待 `fac(1)`
3. `fac(1)` 入栈，达到基例返回 1
4. `fac(2)` 恢复，计算 `2 * 1`
5. `fac(3)` 恢复，计算 `3 * 2` fileciteturn6file0

这说明：

- 递归不是“神奇跳转”
- 而是靠栈帧把“尚未完成的工作”层层压住
- 等更小子问题做完后，再层层回溯

---

### 4.2 `fib(n)` 为什么更占时间

课件紧接着展示了：

```cpp
int fib(int n) { return (n < 2) ? n : fib(n - 1) + fib(n - 2); }
```

这里的问题不是栈深度特别夸张，而是：

- `fib(n-1)` 和 `fib(n-2)` 都会展开
- 子问题大量重叠
- 调用树规模指数爆炸

所以它的主要瓶颈是**时间上的重复计算**，而不是单纯的栈空间。 fileciteturn6file0

---

### 4.3 递归空间复杂度真正取决于什么

课件第 14 页讲得非常好：

> 递归算法所需的空间，主要取决于递归深度，而非递归实例总数。 fileciteturn6file0

这句话很关键。

比如：

- `fib(n)` 的递归实例总数很多，但栈深度其实是 `O(n)`
- 某些单支递归虽然实例总数不算多，但若深度很大，照样占用很多栈空间

所以分析递归空间时，首要看的是：

> **任一时刻调用栈可能达到的最大高度。**

---

## 5. 消除递归：为什么要把递归改写成迭代

### 5.1 动机

课件第 16 页总结得很清楚：fileciteturn6file0

- 递归函数的空间复杂度主要取决于最大递归深度
- 为隐式维护调用栈，需要额外时间与空间
- 为节省空间，可显式维护调用栈，或直接改写为迭代版本

也就是说，消除递归主要有两个方向：

1. **用显式栈模拟隐式调用栈**
2. **彻底改写成循环迭代**

---

### 5.2 `fac` 改写为迭代

课件给出的迭代版是：fileciteturn6file0

```cpp
int fac(int n) {
    int f = 1;
    while (n > 1)
        f *= n--;
    return f;
}
```

这时：

- 时间复杂度仍为 `O(n)`
- 空间复杂度从 `O(n)` 降到 `O(1)`

---

### 5.3 `hailstone` 改写为迭代

```cpp
void hailstone(int n) {
    while (1 < n)
        n = n % 2 ? 3*n + 1 : n/2;
}
```

同样：

- 保留了原算法语义
- 省去了调用栈
- 只用常数空间 fileciteturn6file0

---

### 5.4 `fib` 改写为迭代

课件给出：

```cpp
int fib(int n) {
    int f = 0, g = 1;
    while (0 < n--) {
        g += f;
        f = g - f;
    }
    return f;
}
```

它利用了 Fibonacci 的状态转移只依赖前两个值这一事实，因此可用两个变量滚动维护：

- 时间复杂度：`O(n)`
- 空间复杂度：`O(1)` fileciteturn6file0

这已经不仅仅是“省一点常数空间”，而是渐近意义上的改进。

---

## 6. 尾递归：最容易被消除的递归形式

### 6.1 什么是尾递归

课件定义：

> 在递归实例中，作为最后一步的递归调用。 fileciteturn6file0

例如：

```cpp
fac(n) {
    if (1 > n) return 1;
    return n * fac(n - 1);
}
```

严格地说，这里最后一步表面上是乘法，不是递归调用本身；  
但课程强调这不是本质问题。核心是：

- 当前函数除了等待子调用返回后做一个非常局部的组合外，没有额外复杂控制流程

---

### 6.2 尾递归为什么容易转迭代

课件指出其性质： fileciteturn6file0

- 一旦抵达递归基，便会引发一连串 return
- 调用栈会连续 pop
- 返回地址通常相同或逻辑上可统一
- 因此很容易改写成迭代形式

很多编译器甚至可以做**尾调用优化（tail call optimization）**。

---

### 6.3 `fac` 的尾递归消除过程

课件第 21 页把这一过程写得很漂亮：fileciteturn6file0

1. 原始递归版：

```cpp
if (1 > n) return 1;
return n * fac(n - 1);
```

2. 用显式变量记录子问题解：

```cpp
int f = 1;
next:
if (1 > n) return f;
f *= n--;
goto next;
```

3. 再自然改写成 `while`：

```cpp
int f = 1;
while (1 < n)
    f *= n--;
return f;
```

所以尾递归消除往往有一个统一套路：

- 把“递归参数变化”改成循环变量更新
- 把“返回值组合”改成迭代累积变量

---

## 7. 栈的第一个经典应用：进制转换

### 7.1 问题

课件提出问题：

> 给定一个十进制非负整数，将其转换为任意基数 `base` 的表示。 fileciteturn6file0

例如：

- `12345(10) = 30071(8)`

---

### 7.2 短除法的原理

进制转换的核心是“整除留余”：

- 当前数 `n` 除以 `base`
- 商继续参与下一轮
- 余数就是当前最低位数字

不断重复后，就能从低位到高位得到所有数位。  
课件第 24 页用短除法图清楚展示了这一点。 fileciteturn6file0

---

### 7.3 难点：输出顺序反了

因为余数是：

- 先得到最低位
- 后得到最高位

但人类读数时希望：

- 先输出最高位
- 再输出最低位

这就出现了一个天然“逆序输出”的需求。

---

### 7.4 为什么栈正好合适

课件第 25 页明确指出：fileciteturn6file0

- 位数 `m` 事先不确定
- 希望空间按需增长
- 又希望把“后得到的高位”先输出

如果用栈：

1. 每次求得一个余数，就入栈
2. 最后不断出栈输出

刚好实现了：

- 计算顺序：低位 → 高位
- 输出顺序：高位 → 低位

---

### 7.5 代码实现

课件给出的代码为：fileciteturn6file0

```cpp
void convert(Stack<char>& S, __int64 n, int base) {
    char digit[] = "0123456789ABCDEF";
    while (n > 0) {
        S.push(digit[n % base]);
        n /= base;
    }
}
```

调用时：

```cpp
Stack<char> S;
convert(S, n, base);
while (!S.empty()) printf("%c", S.pop());
```

---

### 7.6 复杂度分析

设转换后有 `m` 位，则：

- 入栈 `m` 次
- 出栈 `m` 次

总时间复杂度：

\[
O(m)
\]

空间复杂度：

\[
O(m)
\]

这里 `m` 大致为：

\[
m = \Theta(\log_{base} n)
\]

## 8. 栈的第二个经典应用：括号匹配

### 8.1 问题观察

课件从表达式中的括号是否匹配出发，例如：

- `(... ) * 2` 这种是匹配的
- 少一个左括号或右括号就会失配 fileciteturn6file0

并指出一个关键观察：

> 除了各种括号，其余符号都可以暂时忽略。 fileciteturn6file0

这说明括号匹配本质上是一个**结构问题**，与具体变量名、数字无关。

---

### 8.2 由外而内不容易，为什么要由内而外

课件先尝试用“减治 / 分治”的外向思路，但指出这些性质大多只给出充分性而非必要性，因此不易直接化为高效算法。 fileciteturn6file0

然后转而提出一个更关键的必要条件：

> 消去一对紧邻的左右括号，不影响整体匹配性。  
> 即：`L ( ) R` 匹配，当且仅当 `L R` 匹配。 fileciteturn6file0

这就把问题转成了：

- 顺序扫描
- 动态维护尚未匹配的左括号

这正是栈的天然任务。

---

### 8.3 一种括号时的算法

算法流程：

1. 从左到右扫描表达式
2. 遇到左括号 `'('`：入栈
3. 遇到右括号 `')'`：
   - 若栈空，立即失配
   - 否则弹出一个左括号
4. 扫描结束后，若栈为空则匹配，否则失配

课件代码如下：fileciteturn6file0

```cpp
bool paren(const char exp[], int lo, int hi) {
    Stack<char> S;
    for (int i = lo; i < hi; i++)
        if ('(' == exp[i]) S.push(exp[i]);
        else if (!S.empty()) S.pop();
        else return false;
    return S.empty();
}
```

---

### 8.4 为什么算法正确

它维持了一个很简单的不变式：

> 栈中始终保存“已扫描前缀中，尚未被匹配掉的左括号”。

于是：

- 若出现右括号而栈空，说明右括号多余
- 扫描结束后栈非空，说明左括号多余
- 只有全过程都不违反，且最后栈空，才是匹配的

---

### 8.5 只考虑一种括号时，其实计数器就够了

课件第 32 页进一步指出：fileciteturn6file0

> 若仅考虑一种括号，只需一个计数器足矣。

因为这时不需要区分括号类型与嵌套结构，只要：

- 左括号数加一
- 右括号数减一
- 过程中一旦减成负数则失配
- 最终归零则匹配

这说明：

> 栈其实是“计数器”的结构化推广，用于记录更丰富的历史信息。

---

### 8.6 多类括号为什么必须用栈

课件给出反例：

```text
[ ( ] )
```

如果只用计数器：

- 左括号个数与右括号个数仍可能相同
- 但类型和嵌套次序错了

因此只靠计数不能处理多类括号。  
必须记住：

- 最近尚未匹配的左括号是什么类型

这正是栈中“栈顶保存最新待配对左括号”的作用。 fileciteturn6file0

---

## 9. 栈混洗：栈生成排列的能力边界

### 9.1 什么是栈混洗

课件定义：

有输入栈 / 序列 `A`、辅助栈 `S` 和输出序列 `B`，只允许：

- `A` 的顶元素弹出并压入 `S`
- `S` 的顶元素弹出并压入 `B`

如果经过一系列操作后，`A` 的元素全部转入 `B`，则 `B` 是 `A` 的一个**栈混洗（stack permutation）**。 fileciteturn6file0

---

### 9.2 栈混洗不是任意排列

例如输入 `1,2,3`，并不是所有 `3! = 6` 种排列都能用一个栈生成。  
课件指出长度为 3 时恰好少一种：

```text
[3, 1, 2]
```

无法实现。 fileciteturn6file0

---

### 9.3 禁形 312

课件给出一个非常经典的结论：

> 一个排列是栈混洗，当且仅当它**不包含 312 模式**。 fileciteturn6file0

即不存在下标 `i < j < k`，使得输出中出现相对顺序：

```text
k, i, j
```

因为：

- `k` 必须先于 `i,j` 输出，说明它先被压到栈顶后弹出
- 但这会阻塞 `i` 与 `j` 的相对出栈方式
- 从而造成矛盾

这是 Knuth 1968 年的经典结果。

---

### 9.4 如何判断一个排列是不是栈混洗

课件给出几种思路： fileciteturn6file0

1. 直接查 312 模式：`O(n^3)`
2. 用等价条件压缩到 `O(n^2)`
3. 直接模拟栈过程：可做到 `O(n)`

最实用的是第三种：

- 按目标输出顺序逐个尝试
- 若当前需要的元素还未到栈顶，就继续从输入推入
- 若已经在栈中但不在顶端，则失败

---

### 9.5 栈混洗与括号匹配、Catalan 数

课件第 42 页指出一个非常漂亮的对应关系：fileciteturn6file0

- 栈的每次 `push` 对应一个左括号 `(`
- 栈的每次 `pop` 对应一个右括号 `)`

于是：

- 一个合法的栈混洗过程
- 等价于一串合法括号序列

因此长度为 `n` 的栈混洗总数，恰好也是：

> **Catalan 数**

这说明本章几个看似不同的问题，背后其实共享同一组合结构。

## 10. 中缀表达式求值：为什么要用两个栈

### 10.1 问题背景

给定一个语法正确的中缀算术表达式，例如：

```text
(0!+1)*2^(3!+4)-(5!-67-(8+9))
```

要求计算其值。课件把这个问题看作“减而治之”的典型应用：  
不断把可优先计算的局部子表达式先算掉，再用其值替代原子串。 fileciteturn6file0

---

### 10.2 难点：如何高效找到“现在该算谁”

中缀表达式里不能只靠“从左到右”或“先括号再乘除再加减”这种口头规则粗暴处理，因为：

- 括号会强行改顺序
- 优先级不同
- 结合性不同（例如幂运算）
- 一元与二元运算符不同

课件第 47 页强调：

> 仅根据表达式的前缀，不足以确定各运算符的计算次序。  
> 必须获得足够后续信息，才能确定哪些运算符可以执行。 fileciteturn6file0

这就需要一种“延迟缓冲”机制。

---

### 10.3 两个栈的职责

课件给出的主算法使用：

- `opnd`：操作数栈
- `optr`：运算符栈 fileciteturn6file0

其核心思想是：

1. 从左到右扫描表达式
2. 若读到数字，压入 `opnd`
3. 若读到运算符，就与 `optr.top()` 比较优先级
4. 根据结果决定：
   - `<`：当前运算符暂时入栈
   - `>`：栈顶运算符现在该算了，弹出计算
   - `=`：通常表示括号匹配或表达式终止，弹出并继续

---

### 10.4 运算符优先级表

课件第 51 页给出一个二维优先级表：

- 行：栈顶运算符
- 列：当前扫描到的运算符
- 表项取值 `<`、`>`、`=` 或非法空格 fileciteturn6file0

这个表实际上编码了：

- 优先级大小
- 左 / 右结合性
- 括号与终止符的特殊语义

于是整个求值过程，就统一成了一个“查表状态机”。

---

### 10.5 三种比较结果的含义

#### 10.5.1 `<`：静待时机

若 `priority(optr.top(), *S) == '<'`：

- 说明当前运算符优先级更高
- 或遇到左括号等需要延迟的情况
- 因此当前运算符入栈，等待未来更低优先级符号触发它执行

课件第 52～53 页把这种情况称为：

> “静待时机” fileciteturn6file0

---

#### 10.5.2 `>`：时机已到

若结果是 `>`：

- 说明栈顶运算符优先级不低于当前运算符
- 栈顶运算符已经可以立即计算

于是：

1. 弹出运算符 `op`
2. 若是一元运算符（如 `!`），弹出一个操作数计算
3. 若是二元运算符，弹出两个操作数计算
4. 结果重新压回操作数栈

课件第 54～55 页把这种情况称为：

> “时机已到” fileciteturn6file0

---

#### 10.5.3 `=`：终须了断

若结果是 `=`：

- 通常是 `(` 与 `)` 配对
- 或开始 / 结束哨兵 `'\0'` 配对

于是只需把栈顶运算符弹出，并读入下一个字符。课件第 56～57 页称为：

> “终须了断” fileciteturn6file0

---

### 10.6 为什么需要表达式结束哨兵 `'\0'`

课件在主算法里先把 `'\0'` 压入运算符栈。  
这其实非常巧妙：

- 它相当于给表达式加了一个最低优先级终止符
- 保证最终所有残留运算符都会被正确清算
- 且算法能统一以“运算符栈为空”为结束条件 fileciteturn6file0

这和前面链表的哨兵节点有异曲同工之妙：

> **用一个人工加入的边界元素，统一处理尾部情况。**

---

### 10.7 为什么二元运算不能直接写成 `calcu(pop(), op, pop())`

课件第 54 页特别问：  
为何不直接写：

```cpp
opnd.push( calcu(opnd.pop(), op, opnd.pop()) );
```

原因是二元运算有操作数次序：

- 栈顶先弹出的其实是**右操作数**
- 再弹出的是**左操作数**

所以必须写成：

```cpp
double pOpnd2 = opnd.pop();
double pOpnd1 = opnd.pop();
opnd.push(calcu(pOpnd1, op, pOpnd2));
```

否则减法、除法、幂等非交换运算都会出错。 fileciteturn6file0

---

## 11. 逆波兰表达式（RPN）：为什么它天生适合栈

### 11.1 什么是逆波兰表达式

课件定义：

> 逆波兰表达式（Reverse Polish Notation, RPN），也叫后缀表达式（postfix）。  
> 它不使用括号，也能表达带优先级的运算关系。 fileciteturn6file0

例如中缀式：

```text
(1 + 2) * 3
```

其后缀式可以写成：

```text
1 2 + 3 *
```

---

### 11.2 为什么它不需要括号

因为在后缀式中：

- 运算符总出现在其操作数之后
- 扫描到运算符时，所需操作数一定已在前面准备好

于是“优先级”不再靠括号和表格维护，而是直接由表达式顺序编码进去了。  
这正是其最大优点。

---

### 11.3 RPN 的求值算法

课件第 69 页给出标准算法：fileciteturn6file0

1. 建立操作数栈 `S`
2. 逐个读取表达式元素 `x`
3. 若 `x` 是操作数，则入栈
4. 若 `x` 是运算符：
   - 弹出所需数目的操作数
   - 执行计算
   - 把结果再压回栈
5. 最终栈顶（也是唯一元素）就是答案

---

### 11.4 为什么 RPN 求值比中缀求值简单

因为中缀表达式求值要解决：

- 优先级
- 结合性
- 括号配对
- 延迟缓冲

而后缀表达式已经把这些结构信息提前“编译”到序列里了。  
所以求值时：

- 不需要运算符栈
- 不需要优先级表
- 一个操作数栈就够了

这正体现了一个重要思想：

> **把复杂结构提前转成更适合机器处理的形式，后续计算就会大幅简化。**

---

### 11.5 从中缀到 RPN 的自动转换

课件第 74 页指出：  
在中缀表达式求值算法中，只要在适当时机把：

- 读到的数字
- 即将执行的运算符

附加到输出串 `RPN` 中，就能同步完成中缀到后缀的转换。 fileciteturn6file0

也就是说：

- 中缀求值算法
- 中缀转后缀算法

它们本质上是同一套框架的两个视角：

- 一个关注“现在把值算出来”
- 一个关注“把执行顺序编码成后缀串”

---

## 12. 队列：另一种受限序列

### 12.1 什么是队列

课件第 79 页定义：

- 只能在**队尾**插入（`enqueue()` / `rear()`）
- 只能在**队头**删除（`dequeue()` / `front()`） fileciteturn6file0

因此队列遵循：

- 先进先出（FIFO, First In First Out）
- 或等价地，后进后出（LILO）

与栈的差别恰恰在于：

- 栈：插入与删除都在同一端
- 队列：插入和删除发生在两端

---

### 12.2 队列的基本接口

队列的核心接口包括：

- `size()`
- `empty()`
- `enqueue(e)`：入队
- `dequeue()`：出队
- `front()`：查看队首
- `rear()`：查看队尾（若实现提供）

课件同样提到可扩展接口如 `getMax()`。 fileciteturn6file0

---

### 12.3 基于列表实现队列

课件给出：

```cpp
template <typename T>
class Queue: public List<T> {
public:
    void enqueue(T const& e) { insertAsLast(e); }
    T dequeue() { return remove(first()); }
    T& front() { return first()->data; }
};
```

即：

- 列表末端作为队尾
- 列表首端作为队头 fileciteturn6file0

因为双向链表支持：

- 末端插入 `O(1)`
- 首端删除 `O(1)`
- 首端读取 `O(1)`

所以队列核心接口都能做到 `O(1)`。

---

### 12.4 为什么基于向量实现队列不如列表自然

课件把“基于向量实现队列”留作课后思考。  
若用普通动态数组，不做循环数组优化，那么：

- 尾部入队可较自然
- 头部出队却会导致前移后缀，变成 `O(n)`

所以原始向量并不适合直接实现高效队列。  
这也是为什么很多语言的队列底层要么用链表，要么用循环缓冲区。

---

## 13. 队列应用：资源循环分配与银行模拟

### 13.1 Round-Robin 资源分配

课件给出经典轮转调度（Round Robin）思想：fileciteturn6file0

```cpp
Queue Q(clients);
while (!ServiceClosed()) {
    e = Q.dequeue();
    serve(e);
    Q.enqueue(e);
}
```

其核心是：

- 队首客户获得一次服务机会
- 服务后重新排到队尾
- 周而复始

这非常适合：

- CPU 时间片轮转
- 打印机排队
- 简单公平调度

它体现了队列的核心优势：

> **天然维护先来后到的公平性。**

---

### 13.2 银行服务模拟

课件还给出一个多窗口排队系统：fileciteturn6file0

- 有 `nWin` 个窗口
- 每个窗口有自己的队列
- 新顾客总加入当前最短队列
- 每个时间单位，窗口服务其队首顾客
- 若服务完成，则该顾客出队

这说明队列不仅是抽象结构，也是一类典型“事件推进系统”的自然模型。

## 14. 双栈当队：一个非常经典的摊还分析例子

### 14.1 基本思路

课件第 87 页给出用两个栈实现队列：fileciteturn6file0

- `R`：负责接收入队元素
- `F`：负责提供出队元素

规则是：

```python
enqueue(e): R.push(e)

dequeue():
    if F.empty():
        while !R.empty():
            F.push(R.pop())
    return F.pop()
```

---

### 14.2 为什么这样就能实现 FIFO

设元素按 `1,2,3,4` 顺序入队：

1. 全部进入 `R`，栈顶是 `4`
2. 当第一次需要出队时，把 `R` 全部倒入 `F`
3. 顺序变成：
   - `F` 栈顶是 `1`
   - 再往下是 `2,3,4`

于是从 `F` 弹出时，顺序正好恢复成：

```text
1, 2, 3, 4
```

这本质上是：

- 一次逆序
- 再利用栈的逆序效果恢复 FIFO

---

### 14.3 最坏情况与分摊复杂度

课件指出：

- 单次最好：`O(1)`
- 单次最坏：`O(n)`  
  （当 `F` 为空且要把 `R` 全部倒过去时）fileciteturn6file0

但如果从整个操作序列看，就不该只盯着这一次“昂贵搬运”。

---

### 14.4 记账法（Accounting Method）

课件第 88 页给出的记账法思想是：fileciteturn6file0

给每个新元素存 4 枚硬币：

- 1 枚支付自身入队
- 2 枚支付未来从 `R` 转移到 `F`
- 1 枚支付最终出队

这样每个元素的全部生命周期费用都已预付。  
因此任意长操作序列涉及 `n` 个元素时，总成本是：

\[
O(n)
\]

于是均摊到每次操作就是 `O(1)`。

---

### 14.5 聚合法与势能法

课件后两页还分别用：

- Aggregate method（聚合法）
- Potential method（势能法）

重新证明双栈队列的摊还 `O(1)`。 fileciteturn6file0

这说明“双栈当队”不只是一个技巧题，更是学习摊还分析的标准例子。

---

## 15. Steap 与 Queap：把 `getMax()` 也做进栈和队列

### 15.1 Steap：支持 `push / pop / getMax`

课件把它命名为：

> Steap = Stack + Heap = push + pop + getMax fileciteturn6file0

实现思路是维护两个栈：

- `S`：真实数据栈
- `P`：前缀最大值栈

其中 `P` 的每个位置保存：

> `S` 中对应后缀的最大值。 fileciteturn6file0

于是：

- `getMax()`：直接返回 `P.top()`
- `push(e)`：把 `max(e, P.top())` 压入 `P`
- `pop()`：同步弹出两个栈

因此这三个操作都可做到 `O(1)`。 fileciteturn6file0

---

### 15.2 Queap：支持队列的 `enqueue / dequeue / getMax`

课件给出的 Queap 思路是：

- `Q` 保存真实队列
- `P` 维护与后缀最大值相关的信息

其实现里，`enqueue(e)` 时需要从 `P` 的队尾逆向更新，直到保证某种单调性。最坏情况下这一过程可能是 `O(n)`。 fileciteturn6file0

虽然课件没展开成标准“单调队列”模板，但本质上已经很接近：

> 为了支持 `getMax()`，额外维护一个“候选最大值队列”。

这类结构在滑动窗口最大值、在线优化等问题中非常常见。

## 16. 直方图最大矩形：栈在算法设计中的高级应用

### 16.1 问题定义

给定直方图 `H[0,n)`，每个柱子的高度非负。  
要求找出面积最大的轴对齐矩形。课件第 97 页给出定义与示意图。 fileciteturn6file0

---

### 16.2 暴力思路为什么太慢

若对每个位置 `r` 都去向左、向右扩展，找到它所支撑的最大矩形边界 `s(r), t(r)`，那么直接做往往需要二次时间。课件也指出 brute-force 并不理想。 fileciteturn6file0

---

### 16.3 单调栈思想

课件第 100 页给出关键算法：  
扫描数组时维护一个**高度严格递增**的栈 `S`，栈里存的是柱子的下标（rank）。 fileciteturn6file0

当扫描到新柱子 `H[r]` 时：

- 若它比栈顶高，则直接入栈
- 若它不比栈顶高，则持续弹出，直到栈顶高度小于它

这样做的意义是：

> 每个柱子一旦被弹出，就说明它“向右扩展”的边界已经被当前更矮柱子确定了。

---

### 16.4 两遍扫描求 `s(r)` 与 `t(r)`

课件先给出一种两遍做法：fileciteturn6file0

- 正向扫描求每个 `r` 的左边界 `s(r)`
- 反向扫描求每个 `r` 的右边界 `t(r)`

这样所有最大支撑矩形都可在线性时间内得到。

---

### 16.5 一遍扫描版本

课件第 103 页进一步给出 one-pass 算法：fileciteturn6file0

```cpp
Stack<int> SR;
__int64 maxRect = 0;
for (int t = 0; t <= n; t++)
    while (!SR.empty() && (t == n || H[SR.top()] > H[t])) {
        int r = SR.pop();
        int s = SR.empty() ? 0 : SR.top() + 1;
        maxRect = max(maxRect, H[r] * (t - s));
    }
    if (t < n) SR.push(t);
return maxRect;
```

---

### 16.6 为什么它是 `O(n)`

看起来有个 `for + while` 双层结构，似乎像 `O(n^2)`。  
但其实每个柱子：

- 最多入栈一次
- 最多出栈一次

因此所有 `push/pop` 总次数不超过 `2n`，  
这正是一个典型的**摊还线性时间**过程。课件第 102 页明确写出整体复杂度是 `O(n)`（by amortization）。 fileciteturn6file0

---

### 16.7 这个算法的本质

它本质上在维护：

> 对于当前扫描位置之前，所有“还没被更矮柱子终结”的候选柱子集合。

一旦遇到更矮的柱子，就说明：

- 某些更高柱子再也不可能向右延伸了
- 它们的最大矩形面积就该在此时结算

这正是单调栈算法的核心范式。

---

## 17. 本章主线总结

### 17.1 栈与队列都是“受限序列”

- 栈：同一端进出，LIFO
- 队列：一端进、一端出，FIFO

它们本质上都不是全新结构，而是线性表接口被约束后的特例。

### 17.2 栈天然对应调用栈

函数调用、递归展开与返回，本质上就是“最后进入的调用帧最先恢复”，所以天然服从栈模型。

### 17.3 递归消除的关键是显式维护状态

- 尾递归最容易改写为迭代
- 一般递归则可考虑显式栈模拟隐式调用栈

### 17.4 栈特别适合“逆序恢复结构”

- 进制转换：低位先得，高位后出
- 括号匹配：维护最新未匹配左括号
- 栈混洗：与括号序列、Catalan 数同构
- 表达式求值：延迟缓冲运算符或直接求值 RPN
- 单调栈：维护“尚未失效的候选状态”

### 17.5 队列特别适合公平调度与事件推进

- Round Robin
- 银行窗口模拟
- 服务系统排队

### 17.6 双栈当队与最大矩形都是摊还分析的典型教材

它们都说明：  
虽然单次操作可能很贵，但从整个操作序列看，总体成本仍可控制在线性量级。

> 这一章真正重要的，不只是记住栈是 LIFO、队列是 FIFO，而是理解：当问题中出现“最近未完成状态”“先来后到服务”“延迟决策”“逆序恢复”“单调候选集”这些结构时，栈与队列往往就是最自然、最高效的抽象工具。
