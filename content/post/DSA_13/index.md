# 数据结构与算法笔记（十三）：串

> 本文基于课程第十三章“串”整理。内容主线包括：  
> 串的基本术语与 ADT、模式匹配问题与蛮力算法、KMP 算法及其 `next[]` / 改进 `next[]` 表、BM（Boyer–Moore）算法中的坏字符（BC）与好后缀（GS）策略、Karp–Rabin 指纹与滚动散列，以及最后简要介绍的 Trie / PATRICIA / Ternary Trie。  
> 这一章真正的核心思想可以概括为一句话：  
> **串匹配的本质不是“从头重复试错”，而是尽可能利用已经发生过的成功与失败，把未来不必要的比较统统省掉。**  

---

## 1. 为什么“串”是单独的一章

在前面的章节里，我们已经学习了：

- 向量、列表：一般线性表
- 栈、队列：受限线性表
- 树、图：更复杂的关系结构
- 搜索树、词典、优先级队列：围绕查询与极值维护的抽象

那么为什么“串（string）”还要单独拿出来讲？

因为虽然串在形式上也只是一个线性序列，但它和一般线性表有一个非常鲜明的不同点：

> **串最核心的操作不是按秩访问某个字符，而是按“模式”去访问一整段结构。**

也就是说，对字符串而言，我们最关心的往往不是：

- 第 `i` 个字符是什么

而是：

- 某个模式 `P` 是否在文本 `T` 中出现？
- 出现在哪里？
- 能否快速判断两个串是否相等？
- 能否高效找到前缀 / 后缀 / 子串关系？

这使得字符串算法逐渐形成了自己独立的一套思想体系。

---

## 2. 串的术语与基本操作

## 2.1 串的线性区间表示

课件第 2 页把字符串 `S` 看成一个从 `0` 到 `n-1` 的字符序列：

$
S = S[0,n)
$

并定义了几个最基本的概念：

### 子串 `substr(i, k)`

表示：

$
S.substr(i,k) = S[i, i+k)
$

前提是：

- `0 ≤ i < n`
- `0 ≤ k`
- 且区间不越界

### 前缀 `prefix(k)`

表示：

$
S.prefix(k) = S.substr(0,k) = S[0,k)
$

### 后缀 `suffix(k)`

表示：

$
S.suffix(k) = S.substr(n-k, k) = S[n-k, n)
$

---

## 2.2 一个非常重要的区间视角

课件第 2 页的图示其实非常值得记住：

- 子串是中间一段 `[i, i+k)`
- 前缀是最左一段 `[0, k)`
- 后缀是最右一段 `[n-k, n)`

并且课件还给出一个串联关系：

$
S.substr(i,k)=
S.prefix(i+k).suffix(k)=
S.suffix(n-i).prefix(k)
$

这说明：

> 子串、前缀、后缀三者其实并不是互相独立的概念，而是同一线性区间结构在不同切分方式下的表现。

这个区间视角在后面的 KMP、BM、Trie 等算法里都很重要，因为它让我们总能把“模式的一段”和“文本的一段”用统一的区间方式来比较。

---

## 2.3 字符串 ADT

课件第 3 页列出了字符串抽象数据类型（ADT）的基本接口：

- `length()`：长度
- `charAt(i)`：第 `i` 个字符
- `substr(i, k)`
- `prefix(k)`
- `suffix(k)`
- `concat(T)`：连接两个串
- `equal(T)`：判断是否相等
- `indexOf(P)`

而最值得注意的，是最后一个：

> `indexOf(P)`

因为这正是：

> **模式匹配（pattern matching）**

的入口。

---

## 2.4 为什么这一章后面几乎都在讲 `indexOf(P)`

课件第 4 页举例说明这些操作都很直观，比如：

- `"data structures".length() = 15`
- `"data structures".charAt(5) = 's'`
- `"data structures".prefix(4) = "data"`
- `"data structures".suffix(10) = "structures"`
- `"algorithms".equal("data structures") = false`
- `"data structures and algorithms".indexOf("algorithm") = 20`

然后课件明确说：

> 以下，直接利用字符数组实现字符串，转而重点讨论串匹配算法。 

这句话很关键。

因为它说明本章的重点不是：

- 如何重新实现一遍 `strlen()`、`strcmp()` 之类的基础库函数

而是：

> **如何解决最核心、也最难优化的字符串问题：模式匹配。**

---

# 第一部分：模式匹配问题与蛮力算法

## 3. 模式匹配问题是什么

## 3.1 Text 与 Pattern

课件第 6 页给出最标准的设定：

- `T`：文本串（text）
- `P`：模式串（pattern）

目标是判断：

- `P` 是否在 `T` 中出现
- 若出现，第一次出现的位置是多少

例如：

- 文本：`a man with money is no match against a man on a mission`
- 模式：`match`

问题就是找出 `match` 在文本中的对齐位置。

---

## 3.2 为什么串匹配通常比“两个随机串比较”更值得研究

课件第 6 页特别做了一个概率上的提醒：

若：

- `n = |T|`
- `m = |P|`
- 字母表大小为 `s = |Σ|`

且满足：

$
2 \ll m \ll n
$

那么对于随机 `T` 和随机 `P`，匹配概率近似为：

$
Pr_{match} \approx n \cdot s^{-m} \rightarrow 0
$

也就是说：

- 随机文本与随机模式之间，真正匹配本来就极少
- 真正有意义的测试场景，往往不是“随机模式”，而是：
  - 随机文本 + 来自文本的真实子串模式
  - 或某种高度相关的数据

这说明：

> 串匹配算法必须擅长处理“存在大量局部相似 / 前后缀重复 / 相关性很强”的输入，而不是只在随机模型下好看。

---

## 4. 蛮力策略（Brute Force）

## 4.1 基本思路

课件第 7～9 页给出最直观的策略：

1. 把 `P[0]` 和 `T[i]` 对齐
2. 从左到右依次比较：
   - `T[i+j]` 与 `P[j]`
3. 若发生失配，则将模式串右移一位
4. 再从头比较
5. 直到：
   - 完全匹配成功
   - 或所有对齐位置都试完

这就是朴素串匹配，也叫 brute-force matching。

---

## 4.2 两种等价写法

课件分别给出两个版本的代码。

### 版本 1

第 8 页代码：

```cpp
int match( char * P, char * T ) {
    size_t n = strlen(T), i = 0;
    size_t m = strlen(P), j = 0;
    while ( j < m && i < n )
        if ( T[i] == P[j] ) { i ++; j ++; }
        else { i -= j-1; j = 0; }
    return i-j;
}
```

这个版本的特点是：

- `i` 指向文本当前比较位置
- `j` 指向模式当前比较位置，也可以理解为已经匹配的长度
- 一旦失配，就通过：

```cpp
i -= j-1; j = 0;
```

把文本位置退回到下一轮对齐起点

### 版本 2

第 9 页代码：

```cpp
int match( char * P, char * T ) {
    size_t n = strlen(T), i;
    size_t m = strlen(P), j;
    for ( i = 0; i < n-m+1; i ++ ) {
        for ( j = 0; j < m; j ++ )
            if ( T[i+j] != P[j] ) break;
        if ( m <= j ) break;
    }
    return i;
}
```

这个版本更直接：

- 外层循环枚举每个对齐位置 `i`
- 内层循环比较整个模式串

这两种写法逻辑等价，只是状态维护方式不同。

---

## 4.3 蛮力算法的复杂度

课件第 7 页给出结论：

- 最好情况：

$
\Omega(n)
$

例如一开始每次都在首字符处立刻失配。

- 最坏情况：

$
O(n \cdot m)
$

典型最坏例子是：

- 文本由大量相同字符组成
- 模式也由大量相同前缀组成
- 每轮都要比到最后几个字符才失配

例如：

- `T = 000000000000...`
- `P = 0000x`

这种情况下，每个对齐位置都几乎完整扫描一遍模式串，导致总复杂度退化为 `O(nm)`。

---

## 4.4 蛮力算法真正低效在哪里

如果只看代码，蛮力算法似乎也没做错什么：  
失配了就右移一位，再试一次。

但它最大的问题是：

> **它完全没有利用“已经比较过”的信息。**

具体来说：

- 某次对齐中已经成功比较过的前缀
- 在下一次对齐里又会被重新比较
- 文本指针 `i` 会频繁回退
- 模式串也总是重新从 `P[0]` 开始

于是产生了大量重复劳动。

这正是 KMP 要解决的问题。

---

# 第二部分：KMP 算法

## 5. KMP 的核心直觉：失败也有信息

课件第 11～13 页从“局部匹配”讲起。

## 5.1 当前已经知道什么

在蛮力算法的任一时刻，都有如下不变式：

$
T[i-j, i) = P[0, j)
$

也就是说：

- 文本中刚刚对齐并成功匹配的那一段
- 已经完全等于模式前缀 `P[0,j)`

所以，一旦在 `P[j]` 处失配，我们其实已经掌握了大量信息：

- 前面那一整段 `T[i-j, i)` 的内容已经“确认过”
- 它和 `P[0,j)` 完全一致
- 这些信息绝不应该在下一轮中被白白浪费

---

## 5.2 KMP 想要达到什么目标

课件第 13 页把目标说得很清楚：

1. **文本串 `T` 永远不必回退**
2. 失配时，只更新模式位置 `j`
3. 已确认匹配过的文本区间可被直接接受，不必重比

也就是说，KMP 追求的是：

> **快速右移 + 文本不回退 + 避免重复比较**

这是对蛮力算法的三重优化。

---

## 5.3 关键问题：失配后应把 `j` 调到哪里？

课件第 13 页进一步追问：

- 若失配发生在 `P[j]`
- 应该把 `j` 调成哪个更小的值 `t`？
- 使得下一轮可以立即继续比较，而无需退回文本

这正是 `next[]` 表要回答的问题。

---

## 6. `next[]` 表：把“失配后的预案”事先写好

## 6.1 `next[j]` 的含义

课件第 15 页给出查询表思想：

- 对于模式串中的每个位置 `j`
- 预先构造一个 `next[j]`
- 一旦在 `P[j]` 失配，就把 `j` 改成 `next[j]`
- 然后继续拿同一个 `T[i]` 去比较

因此 `next[j]` 本质上表示：

> **在 `P[j]` 失配时，模式串中下一个“还值得继续对齐的位置”。**

---

## 6.2 数学定义

课件第 22 页给出更形式化的定义：

对于 `j ≥ 1`，令：

$
N(P,j)=\{0 \le t < j \mid P[0,t)=P[j-t,j)\}
$

也就是：

- 在 `P[0,j)` 中
- 找所有既是前缀、又是后缀的长度 `t`

然后定义：

$
next[j] = \max N(P,j)
$

即取其中最大的 `t`。

这意味着：

> `next[j]` 就是 `P[0,j)` 的**最长真前后缀长度**。

---

## 6.3 为什么这就是失配后最合理的回退位置

因为失配前我们知道：

$
T[i-j, i) = P[0,j)
$

若我们想把模式串右移而文本不回退，那么新的模式前缀必须与这段文本尾部对齐。  
而能继续成立的最大对齐长度，恰好就是：

- `P[0,j)` 的最长前后缀匹配长度

因此：

- 右移太少，会与已知失配信息冲突
- 右移太多，会浪费本可保留的已匹配信息

所以 `next[j]` 是最优选择。

---

## 7. KMP 主算法

课件第 17 页给出经典代码：

```cpp
int match( char * P, char * T ) {
    int * next = buildNext(P);
    int n = (int) strlen(T), i = 0;
    int m = (int) strlen(P), j = 0;
    while ( j < m && i < n )
        if ( 0 > j || T[i] == P[j] ) {
            i ++; j ++;
        } else
            j = next[j];
    delete [] next;
    return i - j;
}
```

---

## 7.1 这段代码最值得注意的两个分支

### 匹配成功，或 `j < 0`

```cpp
if ( 0 > j || T[i] == P[j] ) {
    i ++; j ++;
}
```

- 若字符匹配，则同步前进
- 若 `j = -1`，相当于当前连模式首字符也无法继续承接，那就只能让文本前进一步、模式从头重新开始

### 匹配失败

```cpp
j = next[j];
```

这里只回退 `j`，绝不回退 `i`。

这正是 KMP 最本质的区别：

> **文本指针单调前进，模式指针根据 `next[]` 智能回跳。**

---

## 7.2 为什么文本指针永不回退是个大改进

因为很多应用场景中，文本可能来自：

- 顺序存储介质
- 流式输入
- 网络流
- 外存块读取

在这些场景里，“回退文本”会很昂贵，甚至做不到。  
课件第 35 页在 KMP 总结中也特别指出：

> 特别适用于顺序存储介质。

这也是 KMP 在工程中非常重要的原因之一。

---

## 8. `buildNext()`：如何在线性时间构造 `next[]`

## 8.1 递推思想

课件第 24 页用“减而治之”解释：

若已经知道：

$
next[j]
$

那么在考察 `next[j+1]` 时，只需比较：

- `P[j]`
- `P[next[j]]`

并据此决定是否继续向更短前缀回退。

这说明：

> `next[]` 表并不需要对子串做平方级别的暴力枚举，而是能利用前一项结果递推出来。

---

## 8.2 代码实现

课件第 25 页给出：

```cpp
int * buildNext( char * P ) {
    size_t m = strlen(P), j = 0;
    int *N = new int[m];
    int t = N[0] = -1;
    while ( j < m - 1 )
        ( 0 > t || P[j] == P[t] ) ?
        N[ ++j ] = ++t : t = N[t];
    return N;
}
```

这里的变量含义可以理解为：

- `j`：正在构造 `next[j+1]`
- `t`：当前尝试的候选前后缀长度

流程是：

1. 若 `t < 0` 或 `P[j] == P[t]`
   - 说明当前前后缀可再延长一位
   - 则 `j++`, `t++`, 记录 `N[j]=t`
2. 否则
   - 说明当前候选太长了
   - 应把 `t` 缩到 `N[t]` 再试

这本质上就是：

> **在模式串内部，对“最长前后缀”问题再次做一次 KMP 式回退。**

非常优美。

---

## 8.3 构造复杂度为什么是线性

课件第 25、27～28 页的思想可以总结为：

- `j` 只会单调增加到 `m-1`
- `t` 虽然会回退，但总回退次数也受控
- 因而构造 `next[]` 的总复杂度是：

$
O(m)
$

这比朴素地对每个前缀单独求最长前后缀的 `O(m^2)` 好得多。

---

## 9. KMP 的分摊分析：为什么总匹配复杂度是 `O(n+m)`

课件第 27～28 页专门讨论：KMP 真的比蛮力算法有渐近改进吗？

## 9.1 一个容易让人误解的地方

表面看，某个 `T[i]` 似乎可能被反复拿去和很多个 `P[j]` 比较：

- 先和 `P[j]`
- 再和 `P[next[j]]`
- 再和更短前缀……

这看起来似乎仍可能达到 `O(nm)`。

---

## 9.2 课件中的关键势函数

第 28 页给出一个非常漂亮的量：

$
k = 2i - j
$

并指出：

- 每次成功匹配：`i++`, `j++`，故 `k` 增加 1
- 每次失配回退：`j = next[j] < j`，故 `k` 至少增加 1

因此，**每一次 while 循环都会使 `k` 单调增加至少 1**。

而初始时：

$
k = 0
$

结束时有：

$
k = 2i - j \le 2(n-1) - (-1) = 2n - 1
$

因此循环总步数不超过 `2n-1`，于是匹配阶段复杂度为：

$
O(n)
$

再加上构造 `next[]` 的 `O(m)`，总复杂度即：

$
O(n+m)
$

这就是 KMP 的经典线性时间结论。

---

## 10. 改进 `next[]`：避免“一错再错”

课件第 30～35 页继续讨论一个很关键的优化。

## 10.1 原始 `next[]` 仍可能重复失败

课件第 30 页给出反例：

某些模式串（例如带大量重复前缀）中，若在 `P[j]` 失配后回退到 `next[j]`，  
可能会发生：

- `P[j]`
- `P[next[j]]`
- `P[next[next[j]]]`
- …

这些字符本身彼此相等，  
而当前 `T[i]` 已经和 `P[j]` 失配，那么拿它去和这些相同字符再比，显然还会继续失败。

这就是课件说的：

> “一错再错” 

---

## 10.2 优化思想

因此，若在构造 `next[j]` 时发现：

$
P[j] = P[next[j]]
$

则与其把 `j` 回退到 `next[j]`，不如直接跳得更远：

$
next[j] = next[next[j]]
$

也就是说：

> 若下一个准备比较的位置字符和当前失配字符相同，那么这一步比较注定还会失败，干脆提前跳过。

---

## 10.3 改进后的构造代码

课件第 32 页给出改进版：

```cpp
int * buildNext( char * P ) {
    size_t m = strlen(P), j = 0;
    int * N = new int[m];
    int t = N[0] = -1;
    while ( j < m - 1 )
        if ( 0 > t || P[j] == P[t] ) {
            j ++; t ++;
            N[j] = ( P[j] != P[t] ? t : N[t] );
        } else
            t = N[t];
    return N;
}
```

这里最关键的一句是：

```cpp
N[j] = ( P[j] != P[t] ? t : N[t] );
```

它正对应上面的“若字符相同则继续压缩跳转”。

---

## 10.4 改进的意义

改进 `next[]` 后：

- 最坏渐近复杂度仍然是 `O(n+m)`
- 但会减少很多本来可预见的重复失败比较

尤其对：

- 前缀重复度高
- 字符集较小
- 模式串自相似性强

的情形，常数性能改进很明显。

课件第 35 页总结得很好：

- KMP 既利用了以往成功比对的信息（经验）
- 也利用了以往失败比对的信息（教训）

# 第三部分：BM（Boyer–Moore）算法

## 11. 从右往左比：为什么“尽早试错”更有利

课件第 37～39 页引入 BM 算法。

## 11.1 核心思路：以终为始

蛮力和 KMP 都是从左往右比较。  
BM 的一个重大变化是：

> **每一趟比对都从模式串的末字符开始，自右向左进行。** 

为什么这样可能更好？

因为在很多场景下，失配一旦发生在靠右位置，我们就能一次性排除掉大量对齐方式。  
课件第 38 页甚至用一句简单的话提示：

> `4 + 4 < 12`

意思是：

- 若能更早试错，就不必在中间白白做很多次无效比较

---

## 11.2 BM 的两套策略

课件第 39 页给出 BM 的总框架：

预处理模式串 `P`，构造：

- `bc[]`：坏字符表（Bad Character）
- `gs[]`：好后缀表（Good Suffix）

匹配时：

1. 从右向左比较
2. 一旦失配，利用 `bc[]` 与 `gs[]` 决定模式应右移多少
3. 再继续下一轮

因此 BM 的优势来自两方面：

- 失败提供的“教训” → 坏字符
- 成功提供的“经验” → 好后缀

这与 KMP 很像，只不过方向完全倒过来了。

---

## 12. BC 策略：坏字符（Bad Character）

## 12.1 坏字符的含义

课件第 41 页给出图示。

设当前失配发生在：

$
T[i+j] = X 
e P[j]
$

那么 `X` 就是这次比较中的**坏字符**。

坏字符策略的想法是：

> 既然 `X` 在当前对齐位置和 `P[j]` 不相等，那就把模式串整体右移，让模式中另一个可能匹配 `X` 的位置对准它；若模式里根本没有 `X`，就可以直接跨过更大距离。

---

## 12.2 `bc[]` 表的含义

课件第 43 页给出构造代码：

```cpp
int * buildBC( char * P ) {
    int * bc = new int[ 256 ];
    for ( size_t j = 0; j < 256; j++ ) bc[j] = -1;
    for ( size_t m = strlen(P), j = 0; j < m; j++ )
        bc[ P[ j ] ] = j; // painter's algorithm
    return bc;
}
```

这说明：

- `bc[c]` 记录字符 `c` 在模式串中**最靠右**出现的位置
- 若不存在，则为 `-1`

之所以最后一次赋值覆盖前面位置，课件称之为：

> **画家算法（painter’s algorithm）** 

即最后“刷到”的位置就是最靠右位置。

---

## 12.3 坏字符右移量

失配于 `P[j]` 处时，坏字符策略希望把模式中最靠右的 `X` 移到文本 `T[i+j]` 处，  
因此右移量可写成：

$
shift = j - bc[X]
$

几种情况：

1. **模式中不存在 `X`**  
   则 `bc[X] = -1`，于是：

$
shift = j + 1
$

可一次跳过整段前缀。

2. **模式中存在 `X`，且位置在 `j` 左侧**  
   则右移后正好让该 `X` 对齐。

3. **模式中最靠右的 `X` 竟然在 `j` 右边**  
   那么 `j - bc[X] < 0`，说明坏字符策略本身不应左移，于是通常至少右移 1。

---

## 12.4 BC 策略的优点与极端情况

课件第 45～46 页分析了 BC 的最好和最坏情况。

### 最好情况

若坏字符根本不在模式串中，则一次比较就能排除 `m` 个对齐位置。  
于是课件给出最好情况为：

$
O(n/m)
$

这非常惊人，因为相当于“每比较一次，就跳过整段模式长度”。

### 最坏情况

但若文本与模式高度相似，且每轮都要比到最后才失配，  
那么 BC 可能退化得很接近蛮力算法。

课件第 46 页明确指出：

> 仅靠 `bc[]` 只利用了失败比对提供的信息（教训），还不够。 

于是需要 GS 策略。

---

## 13. GS 策略：好后缀（Good Suffix）

## 13.1 好后缀的直觉

课件第 48 页举例说明：

若当前这一趟从右向左比较，虽然最终失败了，但已经匹配成功了一个后缀，例如：

$
U = 	ext{ATCH}
$

那么这些成功比较本身也是信息。  
既然我们已经知道文本当前末尾有个后缀等于 `ATCH`，就没必要在下一轮把这段信息丢掉。

因此 BM 希望：

> 把模式串整体右移，使得模式中的另一个同样能匹配这个“好后缀”的位置重新对齐。

这和 KMP 利用“已成功前缀”的思想是平行的，只是方向反过来了。

---

## 13.2 GS 的两类情况

课件第 49～51 页给出两种情况。

设当前失配时，已经有一个好后缀：

$
U = P(j,m)
$

### 情况一：模式中存在另一个内部子串 `V(k)` 与 `U` 匹配

那么可选择最靠右的这种出现位置，让它与文本中的 `U` 对齐。  
这样右移最小，但仍能保留已知成功匹配的信息。

### 情况二：模式内部不存在完整的另一份 `U`

那么就退而求其次：

- 在模式前缀中找一个最长前缀
- 它与 `U` 的某个后缀匹配

也就是说，让好后缀的一部分继续发挥作用。

这与 KMP 的“最长前后缀”思想非常相似，只不过对象变成了后缀。

---

## 13.3 `gs[]` 的意义

因此 `gs[j]` 可理解为：

> 当失配发生在 `P[j]` 时，基于当前好后缀应右移的最佳距离。

课件第 54～56 页说明：

- 可先构造一个 `ss[]` 表（suffix size / longest suffix match）
- 再由 `ss[]` 推出 `gs[]`

并指出若暴力构造 `ss[]` 需 `O(m^2)`，  
但通过逆向扫描可做到：

$
O(m)
$

---

## 13.4 BM 的综合性能

课件第 58 页总结 BM 算法：

- 额外空间：

$
O(|\Sigma| + m)
$

- 预处理：

$
O(|\Sigma| + m)
$

- 查找：
  - 最好：

$
O(n/m)
$

  - 最坏：

$
O(n + m)
$

并且课件强调关键因素在于：

- 单次字符匹配成功的概率
- 通常近似：

$
Pr = 1/s
$

这说明 BM 特别适合：

- 字母表大
- 模式较长
- 随机性强
- 失配容易尽早发生

的场景，例如 ASCII / Unicode 文本搜索。

而在：

- 字母表很小（如二进制串、DNA）
- 匹配成功率高

的场景下，BM 的优势会减弱。

# 第四部分：Karp–Rabin 算法

## 14. “串即是数”：把子串比较转成数值比较

课件第 59～63 页引入 Karp–Rabin（KR）算法。

## 14.1 基本想法

若把一个字符串视作某个进制下的数字，那么：

- 模式串 `P`
- 文本中长度为 `m` 的每个窗口

都可对应一个数值指纹（fingerprint）。

于是：

- `P` 在 `T` 中出现  
  必然意味着某个长度为 `m` 的子串与 `P` 数值相同

例如课件第 63 页写道：

- 十进制串可直接视作自然数
- 更一般地，对字符集做编码后，任意长度为 `m` 的字符串都对应一个 `d` 进制自然数

这就把“串匹配”改写成了：

> **滑动窗口指纹匹配问题**

---

## 14.2 为什么不能直接用精确大整数

课件第 65 页指出问题：

若：

- 字符集较大
- 模式串较长

则其对应数值会非常大。  
例如 ASCII 下每字符约 7 比特，只要模式长于 9，指纹就可能超过 64 位。

这样：

- 存储不便
- 比较不再是 `O(1)`
- 每次窗口指纹计算也会退化到 `O(m)`

总体复杂度又会回到蛮力级别。

---

## 14.3 散列压缩：指纹不求绝对唯一，只求快速筛选

课件第 66～67 页给出 KR 的关键技巧：

- 不直接保留完整指纹
- 而是取一个散列值，例如：

$
hash(key) = key mod 97
$

这样：

- 指纹被压缩到机器字长可支持的范围
- 可在 `O(1)` 时间比较哈希值

但要注意：

> `hash()` 值相等，并不是匹配的充分条件。  
> 它只是一个必要条件。 

也就是说：

- 哈希值相等 → 可能匹配，需再做一次严格字符比较确认
- 哈希值不等 → 一定不匹配，可直接排除

因此 KR 是一种：

> **先用哈希快速筛，再用精确比较确认**  
> 的随机化 / 概率化匹配算法。

---

## 15. Rolling Hash：为什么能做到窗口 `O(1)` 更新

课件第 68 页指出 KR 的另一个关键观察：

相邻两个长度为 `m` 的窗口：

- 只差最左一个字符被删掉
- 最右一个新字符被加进来

因此它们的指纹之间存在简单递推关系。  
这和“进制转换”是一回事：

若当前窗口指纹为：

$
x = a_0 d^{m-1} + a_1 d^{m-2} + \cdots + a_{m-1}
$

则下一个窗口只需：

1. 去掉最高位贡献
2. 整体乘 `d`
3. 加上新字符值
4. 再取模

于是每个新窗口的哈希可由前一窗口在 **`O(1)`** 时间更新得到。

这就是所谓：

> **rolling hash（滚动散列）**

也是 KR 算法成立的关键。

---

## 16. KR 与数值计算模型的提醒

课件第 70～74 页插入了一段关于“字宽”的讨论。

这一部分的核心在于提醒我们：

> 在数值算法中，不能一厢情愿地把“大整数运算”也都当作常数时间。

课件通过 `power(a,n)` 的例子说明：

- 若按 RAM 模型的 uniform cost criterion，某些操作似乎是 `O(log n)`
- 但若考虑结果数值本身的位宽，真实成本可能更高

这与 KR 恰好呼应：

- 若不用散列压缩，而真去维护完整大整数指纹
- 则其“看似常数时间”的操作其实并不现实

因此 KR 的精髓就在于：

> **主动把大整数指纹压缩到机器可高效支持的固定字长内。**

# 第五部分：Trie / 键树

课件最后第 75～80 页简要介绍了 Trie。

## 17. Trie 的基本思想

课件第 76 页给出：

> Trie = Digital Tree = Radix Tree = Prefix Tree 

它的本质是：

- 每条边对应一个字符 / digit
- 从根到某节点的路径就对应某个前缀
- 某个完整键（字符串）对应于从根到终止节点的一条路径

因此 Trie 不是基于“字符串整体比较”的结构，  
而是基于：

> **逐字符分支**

来组织词典。

---

## 17.1 为什么 Trie 适合前缀问题

因为在 Trie 中：

- 拥有共同前缀的字符串会共享同一路径前缀

所以非常适合处理：

- 前缀查询
- 自动补全
- 字典词表
- 拼写检查
- IP 路由最长前缀匹配

也就是说，Trie 从另一个角度回答了“模式访问”问题：

- KMP / BM / KR：研究**在一个长文本中找模式**
- Trie：研究**如何组织大量模式 / 词条本身**

---

## 17.2 PATRICIA 与 Ternary Trie

课件最后还提到两种重要变体：

### PATRICIA Tree

通过压缩单分支路径，减少 Trie 中大量只有一个孩子的冗余中间节点。  
适合稀疏键集。

### Ternary Trie

每个节点维护三向分支（小于 / 等于 / 大于），  
可看作 Trie 与 BST 思想的一种混合。

这些内容课件没有进一步展开，但它们说明：

> Trie 家族也是字符串字典结构中非常重要的一支。

---

## 18. 本章主线总结

### 18.1 串的核心操作不是普通按秩访问，而是模式匹配

所以本章虽然起点仍是线性表，但真正难点在于：

- 如何高效处理结构化的“子串相等”问题

### 18.2 蛮力算法的低效来自重复比较

它没有利用：

- 已成功匹配的经验
- 已失败比较的教训

因此最坏达到 `O(nm)`。

### 18.3 KMP 从左向右，靠 `next[]` 利用前缀信息

- 文本指针永不回退
- 构造 `next[]` 只需 `O(m)`
- 整体复杂度 `O(n+m)`

### 18.4 改进 `next[]` 进一步避免一错再错

它利用了“若下一候选字符必然仍会失配，则可直接跳过”的思想，  
主要优化常数性能。

### 18.5 BM 从右向左，综合利用坏字符与好后缀

- `bc[]` 利用失败信息
- `gs[]` 利用成功信息
- 在大字符集、长模式下通常非常快
- 最坏仍可做到线性量级

### 18.6 KR 把串匹配转化为滚动指纹匹配

- 指纹比较快
- 通过 rolling hash 让相邻窗口 `O(1)` 更新
- 哈希冲突需二次确认

### 18.7 Trie 则从“模式组织”角度处理字符串集合

它不再做单次文本匹配，而是：

- 把大量字符串按前缀结构组织起来
- 适合字典与前缀查询问题

---

## 19. 一句话总结

> 这一章真正重要的，不只是学会 KMP、BM、Karp–Rabin 这几个具体算法，而是理解：字符串问题之所以特殊，在于比较的对象不是单个元素而是一整段结构；而高效串匹配的关键，就在于想办法把过去比较中积累下来的经验与教训，转化为下一步更聪明的跳转与筛选。
